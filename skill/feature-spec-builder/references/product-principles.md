# 内置默认产品原则 + 产出前自检 gate + NFR 维度清单

本文件是 feature-spec-builder 的**内置默认产品原则**、**产出前自检 gate 操作细则**与**非功能性需求(NFR)维度清单**。若用户在 SKILL.md 配置区《D. 产品原则》填了自己团队的原则，**以团队的为准**，本文件作补充/兜底；用户没填时直接用本文件这 10 条。

这些原则不是写给客户看的口号，而是**产出特性需求前必须逐条过的一道 gate**（用法见第二节）。它们体现的是「成熟 平台/SDK 产品」区别于「一次性定制功能」的判断：平台的每个特性都要为整个能力体系的长期健康负责，而不只是满足眼前这一个客户。

> SKILL.md 第 7 步只给一句三态摘要 + 指针，gate 的判定细则、红旗清单、冲突优先级**全部集中在本文件**，不在主技能重复。

## 一、内置默认产品原则（10 条）

### 1. 分层能力完整性（Layered Capability Integrity）
**一句话：** 每个特性必须能干净落到某实现层与能力域；落不进、或被迫横跨多层，本身就是设计信号。
**为什么：** 分层模型是平台骨架。归不了类的特性，要么定义不清，要么实际是底层/算法/硬件改动被误装进 SDK 层。
**自检：** 落在单一明确的层与域吗？若跨层，是否已按第 4 步识别每层 owner 与依赖顺序（见 cross-layer-routing.md）？归类含糊处是否打了 [待确认] 而非硬填？

### 2. 单一职责（Single Responsibility）
**一句话：** 一个特性/接口只解决一类问题、只暴露一种关注点。
**为什么：** 把两件事塞进一个接口是 API 蔓延和「谁也不敢动」的根源。
**自检：** 只解决一类问题吗？接口里有没有「顺便」多塞的关注点？该拆成两个特性、或并入某既有能力，而不是新立一个？

### 3. API 预算纪律（API Budget Discipline）
**一句话：** 新增接口前先穷尽「复用/重载/合并现有接口」，能不加就不加；逼近预算上限要预警并论证。
**为什么：** API 表面积是平台最贵的长期负债——每个公开方法都要被永久文档化、测试、兼容。
**自检：** 新增 N 个方法，加之前考虑过用现有接口的参数/重载/组合实现吗？N 是否让总量逼近 [配置区:API 预算]（>80% 要显式预警）？每个新方法都值得占永久预算吗？

### 4. 向后兼容是红线（Backward Compatibility）
**一句话：** 默认不破坏现有调用方；判定为 break 必须配齐迁移路径+弃用窗口+受影响方清单，并升级签字层级。
**为什么：** 平台信任建立在「今天能跑的代码明天还能跑」，一次未经治理的 break 会击穿所有集成商的信任。
**自检：** 是否触碰现有接口/行为/默认值/错误码？若 break：迁移路径、弃用时间表、受影响调用方清单齐了吗？是否标注需 [配置区:破坏性变更签字人] 签字？**除接口签名外**，是否有**示例代码结构 / 对外消息 schema / 文档锚点**已被客户依赖、其变更会 break 下游？若是，同样需迁移路径 + 弃用窗口。

### 5. 最小惊讶 / 开发者体验（Least Surprise / DX）
**一句话：** 命名、参数顺序、默认值、错误信息符合开发者直觉，并与同域既有接口一致。
**为什么：** SDK 的产品质量很大程度上=DX。命名怪异、默认值危险、错误看不懂，即使功能正确也会拉低口碑、推高支持成本。
**自检：** 命名/参数风格与同域既有接口一致吗？**接口/参数名是否遵循全小写下划线（snake_case）、词意完整准确、禁用缩写**（如 `mic`→`microphone`、`cfg`→`config`、`btn`→`button`、`num`→`count`；行业通用且无歧义的除外，如 `id`/`url`/`io`）？零配置默认行为安全合理吗？失败返回的信息能让开发者自己定位根因吗？会不会和某个既有自动逻辑/开关产生「我没料到」的相互作用？

### 6. 安全默认与最小权限（Secure & Least-Privilege Defaults）
**一句话：** 默认配置应是安全那一个；危险/高权限能力默认关闭、按集成商分级授权、必要时加权限门/二次确认/审计。
**为什么：** 平台面对不可控的二次开发者；机器人能物理伤人，把危险能力默认打开等于把安全下限交给最不小心的那个集成商。
**自检：** 默认值偏安全一侧吗（更慢更稳 vs 更快更险）？是否涉及需权限/分级才能用的能力？危险动作（解除急停、解锁速度力矩、直接底盘控制）是否加了门？鉴权失败时是否安全侧失败？是否触及数据/隐私？**若是硬件末端/改装类特性（原型 J）——安全从"危险 API 调用"扩到"危险物理改装"**：承重/力矩/供电红线是否显著警示、超限后果是否写清、禁改区是否标明、改装授权/资质门槛与保修责任/免责声明是否定、资料开放是否按角色分级门控？物理红线给错一个数字就出人身安全事故。

### 7. 可观测且可度量（Observable & Measurable）
**一句话：** 调用方能感知运行状态（查询/事件/日志），平台方能度量使用与健康度（指标/埋点）。
**为什么：** 没有指标的特性是盲飞——平台方不知道它有没有被用、用得对不对、失败率多少。可观测分两个受众，缺一不可。
**自检：** 面向调用方：有状态查询/事件回调/可定位日志吗？面向平台运营：上线后用什么指标判断它健康/被采纳（成功率、调用量、失败分布）？采集方式占位定义了吗（即便细节 [待确认]）？

### 8. 范围纪律与 MVP 切分（Scope Discipline）
**一句话：** 显式声明不做什么；把能力按 P1/P2/P3 切成可独立交付的最小集，先交付有价值的最小切片。
**为什么：** 平台特性最容易「一次想做全」，结果迟迟不能交付或交付一个臃肿难维护的大块。
**自检：** Out-of-Scope 非空且具体吗？P1 构成一个能独立交付、独立验收的有价值最小集吗？有没有把「锦上添花」误标成 P1？若跨层结论为 B/C，P1 是否已收口为本轮真实可交付物（不承诺下层未就绪能力）？

### 9. 可演进性（Evolvability）
**一句话：** 为未来留可向后兼容的扩展形态（options 结构体、预留枚举），但不为臆想的未来过度设计。
**为什么：** 平台接口一旦公开就难改。好接口能在不 break 的前提下长出新能力。
**自检：** 关键入口用了可扩展形态（options/builder 而非长参数列表）吗？有没有为「可能永远不来的需求」过度设计？后续加能力能走兼容扩展而非 break 吗？

### 10. 决策可溯源（Traceable Decisions）
**一句话：** 每个取舍、边界、优先级都要能回答「为什么这么定、依据什么」。
**为什么：** 平台需求会被反复评审、被不同部门挑战，无法溯源的决策会被一遍遍重新争论、在跨部门交接中失真。
**自检：** 所有取舍在《关键决策记录》节里有吗？依据点明来源了吗（客户提出 / 澄清确认 / 团队约定 / **某条产品原则**）？因原则冲突的让步注明让位关系了吗？

## 二、如何把这 10 条当 gate 用（产出前强制执行）

在按《输出模板》产出文档**之前**，对照这 10 条逐条给三态结论：
- **✓ 符合** —— 满足，无需额外动作。
- **⚠ 违背/有取舍** —— 做了妥协。**必须**：①在《产品原则自检》节写清「违背什么、为何不得不违背、取舍是什么」；②在《关键决策记录》节落一行，来源写「产品原则冲突取舍」并注明涉及第几条。
- **— 不适用** —— 与本特性无关（如纯新增查询接口通常与「向后兼容」无关），一句话说明即可。

**轻量特性可压成一行：** 全 ✓/— 时，《产品原则自检》节写「本特性已逐条过 N 条原则，无 ⚠」即可，不必铺 10 行空表；**只有出现 ⚠ 才展开取舍表。**

**这是硬 gate，不是参考清单：** 只要有任一条标 ⚠ 而文档里找不到对应取舍说明 + 决策记录，这份需求就不算「研发能接手」，不得标完成。

**关键防退化红线（与 SKILL.md 第 7 步同一条，故意在参考文件里再写一次，防 LLM 只读到本文件）：** gate 的作用是把不达标项**转成澄清问题或 `[待确认]` 抛回用户拍板**，而非 skill 自行判定通过或自行修复。凡涉及取舍（要不要为兼容牺牲某能力、预算超了砍哪个、危险动作要不要加权限门），必须按《交互式澄清规则》一次一问、带候选，抛回用户拍板，绝不替他决定。

**常见红旗（命中即很可能要标 ⚠ 并抛回用户）：** 为边缘场景新增一串 API（撞预算纪律）；悄悄改了某默认值/错误码（撞向后兼容）；新接口命名/默认值与同域不一致、或用了缩写（撞最小惊讶）；危险能力默认打开（撞安全默认）；一个特性塞两种关注点（撞单一职责）；跨实现层却没拆 owner（撞分层完整性，转第 4 步）；P1 承诺了下层尚未就绪的能力（撞范围纪律 + 假就绪，回写为真实可交付物）；**参考示例/集成/文档类特性未定支持等级**（as-is vs 官方支持）——默认被当成一次性 example，撞范围纪律 + 埋长期负债，必抛回用户拍板。

**冲突优先级（两条硬约束 + 团队可填的其余序）：**
- **硬约束 1：向后兼容是红线。** 默认不破坏现有调用方。
- **硬约束 2：安全默认是红线。** 危险能力默认关闭、安全侧失败。
- **当向后兼容与安全默认彼此相撞，或与其他原则相撞且无明确取舍时——不自动排序，必须升级人工拍板**（按《交互式澄清规则》抛回用户/对应签字人），由配置区 D 组指定的人决定。团队若已在 D 组填了其余原则的相对优先序，按团队序；未填则一律抛回。skill 不为「能物理伤人的产品」预设一个可能误导的全序默认。

## 三、NFR 检查维度清单（填输出模板《非功能性需求(NFR) 与上线就绪》节）

这是与「特性原型」「实现层」正交的第三条轴。**安全与权限、数据与隐私、运营可观测、MVP 切分对每个特性必答**（确实不涉及的可一行带过，但不得留空）；其余按命中填或打 [待确认]。

- **安全与权限**：谁能调、鉴权/分级、危险/高权限动作的权限门与审计、越权后果（必须安全侧失败）。
- **数据与隐私/合规**：采集什么数据（人脸/声纹/语音/轨迹/视频/身份）、存哪、留多久、是否出域/加密/脱敏、是否触发个保法/GDPR/行业合规与同意流程、数据主体删除响应。
- **运营可观测（埋点侧，区别于调用方可观测）**：上线后用哪些指标/埋点看健康度与采纳率、关键告警阈值、出问题如何复现采集现场。
- **性能与容量[按需]**：关键路径目标时延/吞吐/启动 + 测量条件(硬件/负载/场景) + 不达标对策；资源占用(CPU/内存/算力/带宽/电量续航)；与其他特性并跑的资源争用；硬件/固件/平台依赖与可移植。每项给不出硬指标就 [待确认] 并登记「谁、用什么基准、何时定」。
- **灰度与回滚[按需]**：能否灰度、按什么维度放量；特性开关默认开/关；回滚方案与副作用；依赖不满足时优雅降级到什么程度。
- **对既有用户的影响[按需]**：升级后存量集成商行为是否变化、受影响版本范围、弃用周期、是否需通知、兼容垫片。
- **可测试性（验收可执行可复现）**：验收标准能否被可执行验证——测试数据/样本量、前置环境/场景前提、判定方法与工具、通过阈值及来源（谁/何时/方法）。阈值类 [待确认] 必须带「定责人+定法+期限」闭环。
- **优先级与 MVP 切分**：最小可交付集是什么、哪些 P2/P3 可后置、优先级判定依据（客户硬需求/合规必须 vs 增强项）。
- **支持等级与维护承诺（对参考示例/集成/文档类特性——原型 H——必答，其余不涉及一行带过）**：每个交付物是 **as-is**（社区自维护 + 免责声明、不随版本回归）还是**官方支持**（随版本回归/发布、承诺支持窗口）？官方支持则必须配齐"支持哪些版本组合 + 支持窗口 + EOL 跟随策略 + 回归 CI 归属"。极易被默认当"随便给个 example"而埋长期负债。

> 这些维度与 detail-probes 的「特性原型 A~J」、cross-layer-routing 的「实现层 + 外部依赖轴」是正交轴，叠加使用。
