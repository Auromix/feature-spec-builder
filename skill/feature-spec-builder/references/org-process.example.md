# 组织与流程 · 填写范例（EXAMPLE / 示例）

> ⚠️ **本文件是【示例 / EXAMPLE】，不是任何真实公司或产品的数据。**
> 全部角色代号、责任人、厂商名、机型代号、周期天数、人天容量、维护窗口都是**编造的合理示例值**，仅用于演示“一份填好的 `org-process.local.md` 长什么样”。
>
> **两个文件的分工（首次阅读请先看这句）**：本 `org-process.example.md` 是**公开范本**——不含任何真实信息、随 skill 进仓库、只作对照参考；而 `org-process.local.md` 才是你**实际填写、被 skill 读取**的文件——含真名 / 内部链接、必须进 `.gitignore`、绝不提交。简单说：**example 进仓库看，local 留本地填**。
>
> **第三方使用者请把本文件整体替换成你团队的真实信息**，写进你工作目录下的本地文件 `org-process.local.md`。
>
> **隐私红线（务必遵守）**：真实自然人姓名、内部链接（飞书 / 内网 / 工单号 / 短链 token）、真实预算 / KPI / 未公开路线图数字——**只进本地 `org-process.local.md`，加入 `.gitignore`，绝不提交、绝不进任何公开产出**。本 example 文件之所以能公开进仓库，正是因为它里面**没有任何一条真实内部信息**。
>
> 本示例以一家“做机器人二次开发 SDK 的公司”脱敏而来。示例厂商写作 **示例机器人厂商 / Acme Robotics**；示例机型只用中性代号 **R1 / R2 / R3、M-series**；所有链接域名一律用 `developer.example.com` / `example.com`。

## B. 交付流程与关键 gate（5 阶段）

```text
阶段 ①  需求规划与澄清
────────────────────────────────────────────
输入   业务/客户/FAE 汇来的原始诉求
步骤   需求收集 → 需求澄清 → 工作量评估 → 需求锁定 → 需求创建
交付件 PRD（业务或 SDK 产品经理出）
       工作量估分（研发领域代表按“人天”估）
gate   ★ 需求锁定：对齐本版本需求范围；锁定后再改走 CCB（见 D）
责任人 业务/SDK 产品经理（PRD）· 研发领域代表（估分）· 优先级裁决人（冲突裁决）· 项目经理（需求创建）

阶段 ②  需求设计与开发
────────────────────────────────────────────
步骤   方案设计评审 → 日常构建 →（需求变更走 CCB）→ 合入 → 开发 + 自测
交付件 设计文档 + 测试用例（评审通过为准，测试 owner 参与评审）
       自测报告
gate   ★ 方案设计评审：设计文档 + 测试用例齐备且评审通过
       ★ 合入门槛：只进保护分支、必须 Maintainer 审 MR（见 E）
       ★ 自测报告 = 提测入口条件（无自测报告不得提测）
责任人 研发领域代表 + 组件 Maintainer（设计/开发/合入）· 测试 owner（评审参与）

阶段 ③  测试验证
────────────────────────────────────────────
步骤   分支拉取 + 自验证编译 → RC 构建 → 冒烟 → 集成 → 场景 → 压测
交付件 测试报告（含冒烟/集成/场景/压测结论；质量指标按业务线定）
gate   ★ RC 构建通过方可进入系统级测试
       ★ 测试报告结论 = 版本验收/发版评审的质量输入
责任人 测试 owner（场景/嵌入式/集成）· 质量经理（质量结论）

阶段 ④  版本验收与发布
────────────────────────────────────────────
步骤   需求验收 → 发版评审 → 版本说明会 → 版本总结
交付件 需求验收记录 · 发版评审结论 · 版本说明（Release Notes）· 版本总结
gate   ★ 需求验收：由产品/需求提出人逐条验收
       ★ 发版评审：综合质量结论给“发/不发”结论（质量经理 + 产品 + 项目）
责任人 业务/SDK 产品经理（验收）· 项目经理（发版/说明会/总结）· 质量经理（发版结论）

阶段 ⑤  版本生命周期
────────────────────────────────────────────
步骤   导产验证 → 运维例会（周期跟踪现网问题）→ Hotfix（按需增发）→ EOS
规则   Hotfix：仅限“严重且影响场景使用”的问题，评审后增发（见 F）
       EOS：超出维护窗口的老版本原则上不再响应（见 F）
责任人 运维/FAE（导产/现场/例会）· 项目经理 + 质量经理（Hotfix 评审）
```

## C. 版本节奏与管道容量

> 下列数字均为**合理示例值（示例）**，用于演示格式。请按你团队真实节奏替换。

```yaml
版本周期:        6 周            # （示例）一个大版本→小版本节拍
开发时长:        20 工作日        # （示例）除去规划/测试/发布后的净开发窗口
需求评估单位:    人天             # 所有工作量估分统一用“人天”

管道容量:                        # 每版本可接纳的需求总量（投入人力 × 开发天数 × 折算比例）
  大版本:        约 120 人天      # （示例）功能迭代 / 技术创新为主
  小版本:        约 60 人天       # （示例）质量 / DFX / 可维可测可定位为主
  折算说明:      有效人天 = 投入人力 × 开发天数 × 折算比例（示例折算比例 0.6，扣评审/会议/阻塞）
  换算示例:      投入 10 人 × 开发 20 天 × 0.6 ≈ 120 人天（即“大版本约 120 人天”的来历；小版本减半投入即约 60 人天）

大小版本交替:                    # 大、小版本交替排布
  大版本:        功能迭代、技术创新、对外新增能力
  小版本:        质量收敛、DFX、可维护/可测试/可定位性改进

需求锁定时点:    上个版本发版时，锁定下个版本的需求范围   # 锁定后再改走 CCB
```

## D. 变更控制

```yaml
CCB 例会节奏:    每周四上午        # （示例）
CCB 组成:        主持=SDK 线 PjM；成员=SDK 产品负责人 + 平台架构负责人 + 平台 CQE + 相关领域 owner   # （示例）实填替换成真名
适用范围:        需求“需求锁定”后的任何 增 / 删 / 改
决策方式:        以 CCB 议题提报，会上决策通过/驳回/挂起
交付件:          CCB 会议纪要（记录议题、决策、责任人、目标版本）
提报入口:        项目经理汇总议题；发起人需说明变更内容、影响范围、工作量增减(人天)
未过 CCB 的变更:  不得直接改动已锁定的版本需求范围
```

## E. 分支与合入

```yaml
保护分支:                        # 不允许直接 push，一律走 MR
  - develop                     # 集成分支
  - release                     # 发版分支
合入门槛:        必须由对应组件 Maintainer 审核 MR 后方可合入
提测入口条件:    自测报告齐备（无自测报告不受理提测）
自验证:          合入前分支需本地/CI 自验证编译通过
命名建议(示例):  feature/<需求号>-<简述> · fix/<缺陷号>-<简述> · hotfix/<版本>-<简述>
```

---

## F. 版本维护策略

```yaml
维护窗口:        最多维护最新 3 个版本         # （示例）
EOS 原则:        超出维护窗口的更老版本，原则上不再响应（不再修缺陷/不再增发）
Hotfix 条件:     仅当问题“严重且影响场景使用”                # 例：现网导致业务中断、数据错误、安全问题
Hotfix 流程:     现网问题 → 评审（项目+质量）→ 通过后在受影响版本上增发 Hotfix → 回合入主干
版本命名(示例):  vX.Y.Z（X 大版本 / Y 小版本 / Z Hotfix 序号），如 v2.3.0、v2.3.1(hotfix)
```

## 如何据此生成你自己的

你不需要手抄这份示例。有两条路：

**路子一：把资料丢给 skill 自动生成（推荐）**
在支持 feature-spec-builder 的环境里，直接把你公司的**组织/流程/版本管理资料**（职责分工表、人员地图、发版节奏、责任矩阵、分支规范、CCB 规则等）贴给它，并说明“这是我们公司的组织与流程情况，请据此生成/更新我的 `org-process.local.md`”。skill 会把资料**映射**进 A~F 六个分组，抽取后向你复述结构请你确认，已有本地文件时**增量更新、不覆盖你已填的内容**，并对真名/内部链接等敏感项自动走脱敏红线（只落本地、gitignore、不进公开产出）。

**路子二：跑脚手架后手填**
```bash
# 在你的工作目录下执行，生成一份待填的本地配置
bash scripts/init-org-config.sh
# 生成 org-process.local.md（内容为空模板）后，
# 对照本 example 文件，把 A~F 各组替换成你团队的真实值。
```
脚本会创建 `org-process.local.md`（空模板）；该文件名已被仓库根目录的 `.gitignore` 覆盖，因此默认不会被提交。填写时对照本 example 的每个分组照葫芦画瓢即可——把角色代号换成真名、把示例数字换成你团队真实的周期/容量/维护窗口。若你的目录尚无该 `.gitignore` 条目，请先自行加上 `org-process.local.md` 再填写，避免真实信息被误提交。

**填完自检清单**
- [ ] A 表“责任人”列已换成真实姓名/工号（不再是“XX组 owner”这类代号）
- [ ] C 的周期、开发时长、大小版本容量已换成你团队真实值
- [ ] D 的 CCB 例会时间与组成、E 的保护分支名、F 的维护窗口已按实际改写
- [ ] 全文无真名/内部链接残留在会被提交的文件里；真实内部信息只在 `org-process.local.md` 且已 gitignore
