# 产品档案 · 填写范例（EXAMPLE，非真实数据）

> **这是一份【示例 / EXAMPLE】文件，不是任何真实公司或产品的数据。**
> 它是 feature-spec-builder 里 `references/product-profile-config.md` 那份空模板的**完整填好版**，用来给第三方开发者当参照——照着把**你自己产品的真实情况**填进各自本地的 `product-profile.local.md`。
>
> - 本文所有厂商名、机型代号、团队/责任人、域名、预算数字**都是脱敏后的示例值**（多处标了「示例」）。厂商写作中性占位 **示例机器人厂商 / Acme Robotics**，机型只用中性代号 **R1 / R2 / R3**，域名一律 **developer.example.com / example.com**。
> - **第三方使用者应整体替换成自己的真实信息**，别照抄这里的值。
> - **真实内部信息（真名 / 内部链接 / 真实预算 / 未公开路线图）只进你本地的 `product-profile.local.md`，加入 `.gitignore`，绝不提交、绝不进公开产出。** 本 example 文件之所以能公开，正是因为它通篇没有任何真实内部信息。

**本示例的产品形态**：一套**机器人二次开发（二开）SDK / 平台**——脱敏自一个真实机器人二开 SDK，面向**集成商与第三方开发者**，让他们在机器人本体上做二次开发。多机型（R1 / R2 / R3），开发者门户 developer.example.com，二开 SDK 版本随整机版本适配。

## A. 能力层级（对外抽象分几层、各层是什么）

> 本产品对外抽象分 3 层。`L1/L2/L3` 是通用记号，这里对应「原语 / 应用包 / 场景方案」。

```
L1 = 原子能力 / 底层运动·感知·音频接口
       —— 单步、无编排的原语：一次移动、一帧感知、一段音频播放/采集。
          例：move_base_by_distance()、get_rgbd_frame()、play_audio_file()
          （以上函数名仅为演示 D 组命名规范，非真实 API，勿照抄）

L2 = 组合 / 应用级开发包
       —— 把若干 L1 原语编排成可复用的应用能力，带状态与容错。
          例：导航到语义点位、抓取一个已识别目标、一轮语音问答交互

L3 = 完整场景方案
       —— 面向落地场景的开箱方案，含流程编排 + 默认参数 + 验收脚本。
          例：迎宾导览方案、货架巡检方案、front-desk 值守方案

（层数与命名按你产品改；能力型产品常见 2~4 层）
```

## B. 能力域（横向切成哪几个域）

> 本产品横向切成 5 个能力域，用来给需求归类（「这个需求属于哪一块 / 是否跨域」）。

```
域1 = 运动·导航    移动底盘、路径规划、避障、定位建图、多机调度
域2 = 感知         RGBD / 激光 / 相机、目标检测与分割、位姿估计、语义地图
域3 = 交互·语音    麦克风阵列拾音、ASR/TTS、唤醒、对话编排、表情与屏显
域4 = 数据         日志、轨迹与任务记录、点云/图像落盘、遥测上报、数据导出
域5 = 系统·平台    生命周期与状态机、权限与分级、OTA/版本适配、诊断与急停
```

## C. 实现层 × 团队（对内由哪些层承接、各层归谁）

> 这是最需要按产品自定义的一项。能力层级管「对外抽象多高」，实现层管「对内动哪些技术栈、归几个团队」。
>
> **责任人列全部为示例角色代号**（职位 / 组别），**不是真实姓名**。
> **实填时，请把「XX 组 owner」「XX 负责人」替换成你团队里对应的真实责任人姓名**——真名只写进本地 `product-profile.local.md`（gitignore），不要写进任何会公开的文件。

**运行时实现层**（特性真正跑起来要动的技术栈）

| 实现层 | 承接内容 | owner（示例角色代号） | 跨层升级 / 签字人（示例角色代号） |
|---|---|---|---|
| 二开 SDK 层（对外封装） | C++ core + Python 绑定，对集成商暴露的全部接口面 | **SDK 组 owner** | 平台架构负责人 |
| 底层软件（固件·中间件） | 实时控制、驱动、消息中间件、设备抽象 | **底层软件组 owner** | 底层软件负责人 |
| 算法服务（感知 / 规控） | 检测·分割·定位、路径规划、运动规划与控制策略 | **算法组 owner** | 算法负责人 |
| 硬件（机械·电子·末端） | 机械结构、电子/供电、末端执行器（随机型而异） | **硬件组 owner** | 硬件负责人 |

**非运行时实现层**（生态 / 集成 / 文档 / 打包类特性会落这里，别硬塞给核心研发层）

| 实现层 | 承接内容 | owner（示例角色代号） | 签字人（示例角色代号） |
|---|---|---|---|
| 文档 / 开发者门户层 | developer.example.com 的技术资料（quickstart / api / 硬件改装…） | **DevRel / 文档站 owner** | 开发者体验负责人 |
| 构建 / 分发基建层 | SDK 打包、版本适配矩阵、releases 分发、CI | **发布工程 / DevOps owner** | 发布工程负责人 |

```
跨团队升级路径（示例）：
  层 owner → 该层所属组负责人 → 架构评审会（跨层/接口面变更） → 平台架构负责人（拍板）
高危 / 跨安全边界的变更：无论落在哪层，均须走「架构评审会 + 平台架构负责人」双签。
```

## D. 接口与命名约定（对外契约怎么写）

```
命名风格
  = 接口 / 参数一律全小写下划线 snake_case，词意完整，禁用缩写。
    - 禁缩写举例：mic → microphone、cfg → config、pos → position、img → image
    - 参数必须带单位：angle_degrees、distance_meters、speed_mps、timeout_seconds、torque_nm
    - 动词开头表动作：move_base_to_pose()、capture_rgbd_frame()、start_speech_recognition()

语言 / 绑定约定
  = C++ core + pybind11(Python 绑定)。C++ 为唯一事实源，Python 绑定自动生成、签名与之一一对应。
    集成商主要用 Python；性能敏感/嵌入式集成走 C++。

错误与返回约定
  = 失败返回 bool（false=失败），不抛异常；详细原因经 get_last_error() 取错误码 + 文本。
    Python 绑定同样返回 bool，不把 C++ 侧异常透传成 Python 异常。
    （集成商代码在真机现场运行，用返回码而非异常，避免未捕获异常导致进程崩溃。）

物理接口约定[硬件类特性适用]
  = 机械：末端法兰接口 / 孔位阵列 / 公差 / 承重上限 / 力矩上限，须给标注图与配合公差。
    电气：连接器型号 / pinout / 供电电压·电流·极性 / 通信协议(如 CAN / RS-485 / 数字 IO)。
    每个机型的末端硬件不同 → 各出一份《硬件改装资料》，红线（承重 / 力矩 / 供电）单列。
```

## E. 交付形态（产出成什么、跑在哪）

```
形态
  = 仿真 + 真机两者都支持。验收常要求「仿真先行跑通」再上真机
    （高危动作先在仿真里验证，降低真机试错的物理风险）。

支持平台 / 环境
  = 架构：x86_64 + arm64
    OS：Ubuntu 18.04 ~ 24.04（LTS 全覆盖）
    二开 SDK 版本随整机版本适配 —— 每个整机版本对应一条 SDK 兼容线，见门户版本矩阵。

产出输出目录
  = 默认 feature-specs/；每个特性写到  feature-specs/<特性名>/<特性名>.md
```

## F. 用户 / 集成商分级（对谁开放、按级门控）

> 三级递进：封装程度递减、可控粒度递增。**危险 / 高权限动作只对高级别开放**。
> 下面三档名（集成商-入门 / 集成商-进阶 / 集成商-认证）是本示例厂商自起的本地化命名，分别对应模板给的 Associate / Professional / Expert 示例；实填请换成你自己产品的分级名。

| 分级 | 面向谁 | 暴露程度 | 危险 / 高权限动作 |
|---|---|---|---|
| **集成商-入门**（对应模板 Associate） | 入门集成商、方案快速搭建 | 高度封装，只暴露高层能力（L2/L3），参数几乎全默认 | 不开放 |
| **集成商-进阶**（对应模板 Professional） | 有经验的第三方开发者 | 高层入口 + **关键参数可覆盖**，但覆盖值**钳制到安全范围**（超范围自动截断并告警） | 部分开放，需显式开启 + 二次确认 |
| **集成商-认证**（对应模板 Expert） | 深度定制 / 厂商认证集成商 | 细粒度控制，可触及 L1 原语与底层参数 | 开放，但仍受 H 组门控（授权 + 门控开关） |

```
分级差异落地：
  - 同一能力，集成商-入门 只有一个「一键」入口；集成商-进阶 多出可选参数（钳制）；集成商-认证 暴露完整参数与原语。
  - 覆盖值钳制：集成商-进阶 传入越界值 → 截断到安全边界 + 记 warning，不直接失败。
  - 危险动作（见 H）无论分级，都还要过 H 的授权 + 门控，不因是 集成商-认证 就默认放开。
```

## G. API 预算（接口规模纪律）

```
API 预算
  = 对外方法软上限约 250 个（示例值）。这是「控制对外接口面别失控」的纪律，非硬性拒绝。

占用判定
  = 新增每个对外方法（含 Python 绑定暴露的）计入预算。
    - 未超阈值：正常评审合入。
    - 触及 / 超过软上限：触发【接口评审】—— 必须先论证「不能复用现有接口 / 不能合并入参」，
      由平台架构负责人 + SDK 组 owner 会签后方可新增；鼓励用「参数扩展」而非「新方法」。
```

## H. 产品原则取舍（产出前自检 gate 的尺子）

```
采用内置默认原则：是（10 条全文见 references/product-principles.md，均领域中立）

团队自定义 / 补充原则（与内置默认合并，安全类冲突以本组为准）：
  【安全补充 —— 因为机器人能物理伤人，以下高危动作默认关闭、需授权、加门控开关】
    1. 解除急停（release_emergency_stop）
    2. 解锁速度 / 力矩上限（越过出厂安全钳制）
    3. 直接底盘 / 关节控制（绕过规控层的裸运动指令）
    4. 越安全围栏 / 虚拟墙（disable_safety_boundary）
    5. 硬件末端改装触及承重 / 力矩 / 供电红线（越过《硬件改装资料》标注的物理上限）
  上述动作：默认 OFF；需显式授权（分级 ≥ 对应级别）；须经门控开关二次开启；操作留痕。

冲突优先级：
  两条硬约束不可覆盖 —— 【安全默认】与【向后兼容】均为红线；二者相撞必须升级人工拍板，不自动排序。
  其余原则相对优先序：安全 > 向后兼容 > 接口一致性 > 易用性 > 新增能力覆盖面。

破坏性变更签字人：
  = 平台架构负责人（示例角色代号；实填替换成你团队真实责任人姓名，且只写进本地档案）。
```

## 如何据此生成你自己的

这份 example 是给你**对照**用的。要产出你自己的 `product-profile.local.md`，两条路任选：

1. **让 skill 帮你生成（推荐）**
   直接调用 feature-spec-builder skill，把你产品的真实资料丢给它——产品白皮书 / 能力清单 / 架构图 / 模块与团队划分 / 接口文档或 API 列表 / SDK 分层说明 / 集成商等级说明 / 命名规范等——并说明「这是我们的产品 / 情况」。skill 会把这些资料**映射**进模板的 A~H 各组，抽取后**向你复述结构请确认**，没有本地档案就先建、已有就增量更新（不覆盖你已填的）。
   - 敏感内容（未公开路线图、内部成本、个人隐私）skill 不纳入。
   - 全程脱敏红线：真实接口清单 / 团队真名 / 真实预算只落本地文件、`.gitignore`、绝不提交。

2. **先跑脚手架再手填**
   ```bash
   ./scripts/init-product-profile.sh
   # 默认在当前目录生成 ./product-profile.local.md（已存在则拒绝覆盖）
   ```
   然后对照**本 example**逐组把 A~H 填成你产品的真实值：把示例厂商 / 机型 / 域名 / 预算换掉，把「XX 组 owner」「平台架构负责人」等示例角色代号换成你团队的真实责任人。

**填完务必确认**：`product-profile.local.md` 已在 `.gitignore` 中、且从未被提交。真实内部信息只活在本地这一份里——公开产出（跨公司传阅的特性文档）中的实现层 owner 用角色名 / 占位，不带内部团队真名。
