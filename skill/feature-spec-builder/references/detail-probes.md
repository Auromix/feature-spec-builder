# 功能细节探针库（第二层澄清用）

第二层澄清要因特性类型而异。先判断目标特性属于下面哪个原型（可跨多个），再调对应探针清单逐个问。探针是"该问什么"的提示，仍按交互式澄清规则一次问一个、阻塞优先、带候选答案。

通用的九个细节维度见 SKILL.md《第二层 · 功能细节维度》；本文件是把这些维度落到具体特性类型上的问题示范。

---

## 原型 A · 运动 / 导航类
例：导览、巡检、跟随、回充、点到点移动

- 路径不可达时怎么办？（中止上报 / 等待重试 / 就近停靠）
- 途中遇到动态障碍：绕行重规划，还是停下等待？等待多久超时？
- 定位丢失（打滑、遮挡）如何处理？是否需要重定位流程？
- 到达判定标准是什么？位置误差阈值（cm）、朝向是否要求？
- 行进速度、避障安全距离：默认值、可调范围、谁能调？
- 途经点是预先地图标注，还是运行时可动态增删？
- 多个移动任务并发时如何调度？是否排他？
- 急停 / 人为接管后，任务状态如何收尾？能否恢复？

## 原型 B · 感知 / 识别类
例：人形检测、物体识别、人脸、场景理解

- 误检 / 漏检如何处理？是否有置信度阈值？默认多少、可调吗？
- 多目标同时出现：全返回、返回最优、还是按规则筛选？
- 输出坐标用哪个坐标系？相机系 / 机体系 / 世界系？
- 检测频率 / 延迟要求？实时流还是单帧请求？
- 光照、遮挡、距离的有效工作范围边界在哪？
- 无目标时返回空集还是报错？调用方如何区分"没检到"和"没在跑"？
- 结果是否需要带时间戳 / 帧号以便对齐其他数据？

## 原型 C · 人机交互 / 对话类
例：语音播报、问答、多轮交互、引导对话

- 用户中途打断怎么处理？打断后状态如何？
- 无响应超时多久？超时后的行为（重复提示 / 退出 / 转人工）？
- 多轮上下文要保留多久？跨会话是否保持？
- 并发多人说话 / 多路输入如何取舍？
- 内容由谁提供（本特性 vs 集成商传入）？边界在哪？
- 失败兜底话术 / 降级路径是什么？

## 原型 D · 数据 / 查询类
例：状态查询、列表、历史记录、统计

- 返回数据的 schema / 字段定义？
- 数据量大时是否分页？分页参数、默认页大小？
- 空结果如何表达？
- 排序 / 过滤条件支持哪些？
- 数据时效性：实时、缓存（多久）、还是快照？
- 权限：不同集成商分级看到的数据范围是否不同？

## 原型 E · 长时 / 异步任务类
例：建图、批量处理、训练、长流程编排

- 进度如何上报？百分比 / 阶段事件 / 轮询？
- 能否取消？取消后中间产物如何处理？
- 中断后能否断点恢复，还是必须重来？
- 预计耗时 / 超时上限？超时后行为？
- 任务结果存哪、保留多久、怎么取？
- 同类任务能否并行，还是必须串行排队？

## 原型 F · 标定 / 配置类
例：参数标定、设备配置、能力开关

- 配置在何时生效？立即 / 重启后 / 下次任务？
- 非法配置如何校验与拒绝？是否回滚到上一个有效值？
- 是否需要持久化？掉电后是否保留？
- 默认配置是什么？恢复出厂如何触发？
- 配置变更是否影响正在运行的任务？

## 原型 G · 接口暴露 / 底层能力封装类
例：把已有固件/底层能力开放到 SDK、给现有引擎加一对 set/get、薄封装透传

这类特性"形状"往往很小（常只 1~2 个方法），第二层**不必硬套完整 F+D 九维**，核心就两件事——**核实底层、定死契约**：

- **底层到底支不支持？**（最高优先、最易翻车）"固件已支持、只是没暴露"是否经底层团队确认？若实为"从零新建能力"（如底层根本没有波束成形），工作量与风险是另一个量级，必须先钉死再谈接口。
- **透传还是适配？** SDK 层是原样透传底层接口，还是要做单位／坐标系／枚举的转换？转换规则写清。
- **契约四要素**：输入（类型/单位/范围/坐标系/默认值）｜输出｜**非法输入的返回与副作用**（拒绝返回错误 vs 钳制 vs 就近吸附，且失败时原值是否保持）｜同步/异步。
- **标称范围 vs 实际能力边界**：对外声明的范围与底层真实可达的分辨率/档位常常不一致（如波束角实际只可分辨有限档）。决定对外如何呈现：声明全范围+内部就近映射，还是明示有限档位。
- **生效与持久化**：立即生效还是需重启？掉电是否保留？默认值是什么？
- **与既有上层逻辑的耦合**：这个 set/开关会不会和已有的自动逻辑打架？多处调用按 last-writer-wins 还是互斥？
- **可观测性最小集**：有没有配套的 get/查询让调用方确认当前状态？失败仅靠 bool 够不够，还是要错误原因——可先 bool、把"细化错误码"挂待确认。

**G 的跨层级联（当「底层支不支持」答「不支持/不确定」时，别停，往下追到触底）**

原型 G 的头号问题「底层到底支不支持？」一旦答出「不支持」或「不确定」，本特性就从「二开薄封装」升格为**跨层需求**——立即转入 `references/cross-layer-routing.md` 的跨层分解，按层级联追问：
- **追固件层**：是固件**缺接口暴露**（开个口，工作量小），还是**根本没实现**（从零做能力，量级差一个数量级）？必须找固件 owner 部门确认，不能替答。
- **追算法层**：若能力本质是算法（波束成形、降噪、SLAM、标定），算法是否已就绪？是否需新训练/标定数据？归算法组。
- **追硬件层**：能力是否受硬件形态物理限制？（如波束成形依赖麦克风阵列是几元/线阵/环阵；某些角度物理做不出）归硬件组——硬件改动周期最长，最该早识别。**一旦触到物理不可行，立即标「非二开层可独立交付」并停止细化二开层接口签名**。
- 每往下一层就把该层标进跨层分解表，记清 owner 与「阻塞上层」关系，并按跨部门待确认项的三元组（验证 owner｜评审节点前给答案｜未答复默认假设）登记。

**对客户呈现口径（必须定责，不是技术细节）：** 本原型《标称范围 vs 实际能力边界》一旦不一致，**这个对外口径必须有人拍板**——决定「声明全范围+内部就近映射(须告知会吸附到最近档)」还是「明示有限档位」，记入《关键决策记录》注明拍板人；且验收标准必须按**实际能力边界**写，否则客户按标称范围(连续可调)验收、实际被吸附到档位，验收直接判不通过，变成验收/口径纠纷。

> 示例（set_microphone_beam_angle）：二开层确认「只是透传」→ 追固件「波束成形固件支持吗？」→ 若否，追算法「波束成形算法就绪吗？」→ 再追硬件「当前麦克风阵列支持波束成形吗(几元/阵型)？」。三问下来，一个「加一对 set/get」的小需求暴露出跨 2~4 层、跨 3~4 个部门的真相。完整端到端跑法见 cross-layer-routing.md 末尾示例。

## 原型 H · 生态集成 / 参考实现 / 开发者文档类
例：SDK↔ROS/ROS2 桥接示例、第三方框架适配包、Docker/K8s 部署样例（部署样例归 H；实际部署/分发包归 I）、SDK 语言绑定、quickstart/样例工程、官网文档板块

> **H↔I 判别（易混，先分清）**：交付物是**给客户照着扩的示例/适配代码/文档**（客户要仿照）→ 原型 H；交付物是**可安装的包/安装脚本本身**（客户要装到机器上）→ 原型 I。二者常同现（如 SDK 语言绑定 = 示例(H) + 预编译离线包(I)），命中即合并探针，但别用 H 的示例防腐/可仿照性去套安装包——会漏掉依赖捆绑/离线校验/权限落点。

这类特性的产物是**示例代码/适配包/文档**，不是新的运行时能力。**底层通常完全支持**（复用已有 SDK 接口），所以别用原型 G 的"底层支不支持"去套——那会把澄清带偏；真正的风险在**支持边界、版本矩阵与维护防腐**。核心 6 问（其余维度交叉引用、不重写）：

- **① 支持等级（最高优先，一句定生死，对标 G 的"底层支不支持"）**：这是 **as-is 社区样例**（不随版本回归、issue 不进 SLA、明示"仅供参考"）还是**官方支持参考实现**（进 CI、随 SDK 每版回归、报 bug 我们修、承诺支持窗口）？两者工作量与 owner 差一个量级，决定后续全部范围——必须先钉死。默认容易被当"随便给个 example"，埋长期负债。
- **② 目标生态/版本矩阵 + EOL**：支持哪些框架/发行版 × OS × 语言运行时 × 构建系统 × 关键依赖（DDS/编译器/ABI）的组合？逐行锁定并标各格 **EOL**（外部生态的 EOL 时钟不归你管，见 cross-layer-routing.md 的外部依赖轴）。矩阵是笛卡尔积，越大 CI/维护越贵；不锁不可验收。
- **③ 交付物形态**：可运行示例仓 / 预编译包 / Docker 镜像 / 纯文档？决定分发与"能不能在客户机器上编过"。
- **④ 示例防腐（anti-rot，此类特性最大长期返工源，别处问不到）**：owner 是谁？示例跟随哪个 SDK/上游版本？**进不进 CI 矩阵构建+冒烟**？SDK 发版是否触发回归？版本追随窗口几个？不定，示例会迅速腐化成编不过的死代码。
- **⑤ 我方维护 vs 客户自持边界**：哪部分我们保证编过/维护，哪部分是"客户按自己环境填"的占位？连带定：build 不过找谁、进不进支持 SLA、示例许可证与免责声明（as-is 尤其要有免责声明，否则与"不承诺"口径法务对不上）。
- **⑥ 可仿照性 / 扩展范式与简单度（参考实现·示例类尤其关键，"参考价值 > 一次性能跑"）**：示例是给客户**照着扩**的，不只是"能跑"。有没有一套**统一、可复制的映射/适配范式**（客户加自己的接口 = 照抄一个单元 + 注册一行，而非改一大坨）？有没有"**如何新增你自己的 X**"的文档 + 一个可跑的加法示例？是否**足够简单、反过度封装**（不引入自研框架/复杂抽象，优先可读可复制）？验收锚点：**一名没读过内部实现的开发者照着能在 N 分钟内新增一个**并跑通。别把参考示例做成一个大而全、客户看不懂也仿不了的黑箱。

> **交叉引用（不在 H 内重写）**：双向数据/消息契约 → 复用原型 D（schema）+ F 与《层间交接契约》的"定义权归属"；文档站 IA 落点/是否平级 → 归"文档/开发者门户"独立特性（见 SKILL.md 第 4 步门户层路由）；采纳度量 → 通用九维第 9 条"面向平台运营的可观测"。
> **原型 H 的"跨"不在技术栈**：天然跨**交付职能**（代码/构建/文档/支持），而非固件→算法→硬件；cross-layer 的下探基本不触发，改用"跨职能 owner + 外部依赖轴"视角（见 cross-layer-routing.md）。文档板块通常拆成独立特性、用关联特性互指。完整范例见 `references/example-ros-integration.md`。

## 原型 I · 打包 / 分发 / 离线安装 / 部署类
例：离线/气隙安装包、一键安装脚本、预编译制品分发、Docker/K8s 部署包、SDK 安装器、私有源/离线源

产物是**"装到目标机器上即用的制品"**——不是给客户照着扩的源码（那是 H），也不是运行期参数（那是 F）。**底层能力已存在**，别用 G 的"底层支不支持"去套（会诱导向固件下探）；核心风险在**制品能不能在目标机器上干净装成、可信、可升可卸**。owner=**构建/分发基建层**（SDK 构建/DevOps）+ 文档门户层，**别默认派二开组**。

**复用 H / 外部依赖轴，不重列**：支持等级（官方支持 vs best-effort）→ H①；交付物形态 → H③；DX 验收锚点 → 改写自 H⑥（"一名无该 SDK 背景的工程师，拿离线包+README，在**断网**目标机上 ≤N 分钟装通并过自检、全程只靠 README 不求助"）；**目标平台矩阵 + EOL** → 复用 H② + cross-layer 外部依赖轴（OS 发行版 × CPU 架构 × 语言运行时 ABI × glibc 基线，逐格锁 + 标 EOL，不锁不可验收）。

**I 独有 5 问（H 问不到）：**
- **① 依赖捆绑策略（离线头号决策，"离线≠给个 tar 包"的分界）**：完全自包含（bundle 系统库/运行时/驱动依赖）vs 假设基础环境 vs 私有离线源 vs 打包隔离环境（容器/可重定位 env）？**含再分发许可可行性前置 gate**——某些依赖（如 CUDA redistributable、GPL/LGPL 静态链接）可能根本不允许打进包，先核"哪些能合法 bundle、哪些必须客户自带"。
- **② 安装形态 + 交互↔静默边界**：pip+本地 wheelhouse / conda-pack / 自解压 shell(.run) / tarball+install.sh / .deb 离线源 / Docker 镜像 tar？"**一行命令(可脚本化)**"与"**交互(需人应答)**"潜在矛盾——气隙客户常要**无人值守批量部署**，必须提供 `--yes`/应答文件的静默模式（但验签/提权不宜静默跳过）。
- **③ 完整性与来源信任（串安全默认红线）**：只给 checksum（防传输损坏），还是加**签名 + 公钥带外分发 + README 强制验签**、且**验签失败 fail-closed 硬阻断安装**？bundle 内第三方依赖的 license 再分发与 CVE/SBOM 谁治理？气隙客户尤敏感——离线包被替换无联网兜底。
- **④ 安装态生命周期（安装态的向后兼容）**：装完怎么**自检**（import + 版本 + native `.so`/ABI 探测）？**重复执行是否幂等**？**原地升级**？**干净卸载/失败回滚**？以及**与既有安装/旧版本的同机共存/冲突**（覆盖/并存/拒绝）？native 扩展（pybind `.so`）按 abi3 跨版本还是每个次版本各一份、不匹配时如何**安全失败**（别抛 C++ 符号错）。
- **⑤ 权限落点（串安全默认，默认偏免 root）**：需要 **sudo/系统级** 还是支持**纯用户级**（venv/用户目录、不污染系统 Python）？装到哪、是否改全局 PATH？危险的系统级写入是否二次确认？

> **失败可诊断（离线唯一 UX）**：离线现场无网，装失败必须**当场给可读根因 + 建议动作**（缺哪个库/Python 版本不符/ABI 不匹配/验签失败/权限不足），并可落一个**可回传的诊断包**（环境探测+校验结果+分级退出码）；README 是离线唯一支持界面，其完整度即验收项。
> **探针产物落点（复用现有模板节，不新增节）**：平台矩阵+捆绑→《依赖与前置》《影响评估》；完整性签名+权限→《NFR·安全与权限》；幂等/升级/卸载/回滚→《影响评估·兼容·发布策略》；失败可诊断+README→《行为规格·可观测性》+ P1 的 DX 验收锚点。完整范例见 `references/example-offline-install.md`。

## 原型 J · 硬件末端 / 物理接口规格与改装资料类
例：末端执行器机械+电气规格书、硬件改装/接口资料、第三方配件对接文档、机械装配与电气接线手册、换件适配 demo

这类特性的**对外契约是物理接口**——机械（安装法兰/孔位阵列/尺寸公差/承重/力矩/惯量）+ 电气（连接器料号/pinout/供电包络/信号协议），**不是软件 API**（别套《接口契约草案》的软件字段——签名/错误码/同步异步在这里不适用）。**底层能力已存在**（是把已有硬件的接口数据开放出来），别套 G 的"底层支不支持"。核心风险：**给错一个数字（承重/力矩/供电/pinout）= 客户把自制硬件接到会动、会伤人的机器人上出事故 + 法律责任**。owner：**内容真相源=硬件/机械/电子团队**（定义权在工程图纸/PCB/BOM），由**文档门户层**承载，**别默认派二开/文档团队去"照二手资料写"**。

**复用 H 骨架、换成物理内容（不重列）**：支持等级/责任边界 ← H①（官方背书数据 vs 参考数据 + 保修/授权/免责）；按型号×硬件版本锁矩阵 ← H② + 外部依赖轴（这里是**硬件版本/批次**而非软件版本）；图纸-实物一致性防腐 ← H④（对象换成工程图纸）；可仿照性 ← H⑥（给"参考末端"的完整机械+电气设计供客户照抄）；文档站板块 IA/是否平级 → 文档门户层独立特性（关联特性互指）。

**J 独有 6 问：**
- **① 物理契约矩阵（最高优先，错一个数字出事故）**：机械——安装法兰/接口标准、螺栓孔位阵列与螺纹、定位销、尺寸与**公差带**、**最大承重（静/动态分开）**、**各轴力矩上限**、质心/惯量、安装姿态；电气——连接器**厂商料号+配对件**、**pinout 逐针脚**（编号↔信号↔电平）、线序、**供电电压/电流/功率包络（含瞬时峰值）**、通信协议（型号+波特率/时序）、接地/屏蔽。**逐格给死、可据以接线与加工验收**。
- **② 型号 × 硬件版本矩阵**：这些参数**按机器人型号 × 硬件版本/批次**锁定（同型号改板会改孔位/pinout）；规格书标**适用硬件版本范围/序列号区间/固件依赖**。不锁版本，客户拿新批次按旧书接线=损坏。
- **③ 图纸-实物一致性与防腐（硬件类最大长期返工源）**：每个数据源头是哪份**工程图纸/PCB/BOM**、谁**背书与实物一致**、图纸改版时规格书**怎么同步**。图纸改了、对外规格书没跟=客户按过时数据加工的配件装不上。
- **④ 物理安全边界与责任（安全红线，务必强制问一次）**：**承重/力矩/供电红线**怎么在文档显著警示 + **超限后果**（触发保护/损坏/失控）；**禁改区**（哪些接口绝不可动、动了会解除急停/安全围栏）；改末端后是否需**负载重标定/安全参数申报**；**改装授权/资质门槛**、**保修/责任边界**（改装是否失保、故障责任）、**免责与合规声明**、**开放范围**（谁能拿这份资料、是否需角色分级/登录门控）。
- **⑤ 交付物精确度与自验**：交付形态（网页/可下载 PDF、**STEP/DXF/CAD 图纸、3D 模型**）；**单位制**（公制/英制）、**公差标注标准**（ISO/ASME）、坐标系与基准面；客户如何**自验"我加工的配件能装上"**（关键配合尺寸检验清单 / 对配样件）。
- **⑥ 可仿照性（换件适配 demo）**：是否给一个**参考末端的完整机械+电气设计 + 换件适配步骤**（机械装配 + 电气接线 + 可能的标定/配置），供客户**照抄改**？demo 形态（图文/代码/实物设计文件）决定 owner。对标 H⑥，但对象是机械/电气设计。

> **接口契约概念一般化**：见 SKILL.md《接口契约草案》——接口分**软件接口**与**物理接口**两类；物理接口特性走本原型 J 的物理契约矩阵字段，不套软件签名/错误码字段。
> **安全一般化**：本原型的安全不是"危险 API 调用"，而是"**危险物理改装**"（见第一层维度 13 与产品原则 6 的物理改装分支）。完整范例见 `references/example-hardware-mod.md`。

## 用法提示

- 一个特性常跨多个原型（如"导览"= A 运动 + C 交互的到点讲解回调，或集成类 H 叠加 G），把相关原型的探针合并后挑关键的问。
- 探针不是逐条必答的清单。对照通用九维，判断哪些细节对**这个**特性是会改变研发实现的关键决策，优先问那些；不关键的留合理默认或打 `[待确认]`。
- **H / I / J 别混**：交付物是"照着扩的源码/文档"用 **H**；是"装到机器上即用的软件制品（离线包/安装器/部署包）"用 **I**；是"**硬件末端/物理接口规格与改装资料**（机械+电气、给客户接自制硬件）"用 **J**——J 的对外契约是**物理接口**（法兰/孔位/承重/力矩/连接器/pinout/供电），不是软件签名，真相源 owner 是硬件/机械/电子团队。三者常同现（如 SDK 语言绑定=示例(H)+离线包(I)；末端改装=规格书(J)+官网板块(文档门户独立特性)+适配 demo(H/J)），命中即合并探针，但别拿 H 的软件探针去套 I/J。**H/I/J 都不做固件/算法/硬件下探**（别套 G 的"底层支不支持"）。
- 问出来的结论填进输出模板《行为规格》节，涉及取舍的记入《关键决策记录》节。
