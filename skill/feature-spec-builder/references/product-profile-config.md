# 产品档案 · 本地配置（模板 + 机制）

## 这是什么、为什么放本地

feature-spec-builder 的方法论本身是**领域中立**的：它不假设你做的是机器人、SaaS、芯片 SDK 还是云平台。但要把一句话需求细化到"研发能直接接手"，skill 需要知道**你的产品长什么样**——它的能力怎么分层、分哪些能力域、对内由哪些实现层承接、接口怎么命名、交付成什么形态、给哪几级用户/集成商用、API 预算多少、产品原则取舍是什么。这套**产品自有的坐标系与约定**每个产品都不一样，而且往往含内部命名/团队/接口清单，所以**不写死进 skill**，而是由你在本地生成一份**产品档案**填自己产品的情况，与 skill **搭配使用**。

- 公开的 skill 里：只有下面这套**产品无关的占位模板**（能力层级、能力域、实现层、命名约定、交付形态、分级、预算、原则），并附**多领域示例**（机器人 / SaaS / 芯片 SDK / 云平台 / 音视频 SDK），**没有任何真实产品内部信息**。skill 正文里出现的机器人/固件·算法·硬件、波束、ROS、末端改装只是**示例领域**，不是前提——它们是好范例，帮助理解方法，但你的产品用**自己的产品档案**接入即可。
- 你本地填好的档案：含你产品真实的层级命名、接口清单、内部团队、预算数字——**本地私有、加入 `.gitignore`、绝不提交**。

> 与 `org-process-config.md`（组织与流程）的分工：**产品档案回答"你的产品长什么样"**（层级/域/实现层/命名/形态/预算/原则）；**组织流程配置回答"你公司怎么协作"**（角色地图/交付 gate/版本节奏/CCB/维护策略）。两份都产品无关、都本地生成、都 gitignore、都可由"描述/丢资料"触发生成，配合使用：一份定坐标系，一份定协作流程。

## 怎么用（三步）

> 📎 **想先看一份填好的完整范例？** 见 [`product-profile.example.md`](product-profile.example.md)——一份**脱敏**的「机器人二开 SDK」产品档案（A~H 全部填满、无任何真实公司信息），照着把你自己产品的真实值替换进去即可。它是本模板的「已填好版」。

1. **生成本地文件**：跑 `scripts/init-product-profile.sh`，或手动把下面《模板》整段复制成你工作目录下的 **`product-profile.local.md`**。
2. **按你产品填**：能力层级填你自己的分层（`L1/L2/L3` 或你产品自己的名字，如 `原语层/服务层/方案层`、`API/SDK/Solution`）、能力域填你产品的域、实现层填**你产品自己的**层（软件 SaaS 可能是 前端/后端/数据/基础设施；芯片 SDK、云平台、音视频 SDK 各不同，**不要硬套机器人的固件/算法/硬件**）、命名约定填你们的接口规范、交付形态/分级/预算/原则同理。
3. **搭配 skill**：细化需求时，skill 会读这份本地档案——用**能力层级 + 能力域**给需求归类、第 4 步跨层分解用**你的实现层**指派 owner、接口契约草案套**你的命名约定**、影响评估按**你的 API 预算**判占用、产出前用**你的产品原则**跑自检 gate。**没有这份档案时**，skill 用内置的通用坐标系占位、把推断值一律打 `[待确认]`、并提示你生成一份。

> **隐私红线**：真实产品内部信息（未公开的接口清单、内部团队名、真实预算数字、真实责任人）只进 `product-profile.local.md`（本地、gitignore）。skill 产出的特性需求文档若要公开或跨公司传阅，实现层 owner 用角色名或占位、不带内部团队真名（除非该文档本身是内部件）。

## 从产品资料自动生成本档案（skill 被触发时怎么做）

当用户**描述自己的产品**，或**丢来产品/架构/接口资料**（产品白皮书、能力清单、架构图、模块划分、接口文档/API 列表、SDK 分层说明、集成商等级说明、命名规范等）并说"这是我们的产品/情况"，这就是**生成产品档案**的信号——不要只当背景读过就算。按下面把资料**映射**进 `product-profile.local.md`：

| 资料里常见的 | 映射到本档案 |
| --- | --- |
| 产品分层 / 能力等级 / 抽象层级（原语·组合·方案 / API·SDK·Solution / L1·L2·L3…） | **A 能力层级** |
| 能力清单 / 模块划分 / 功能域 / 子系统 | **B 能力域** |
| 架构图 / 技术栈分层 / 各模块负责团队 / 子系统 → 部门 | **C 实现层 × 团队** |
| 接口文档 / API 列表 / 命名规范 / 语言绑定 / 错误约定 | **D 接口与命名约定** |
| 交付形态 / 部署形式 / 支持平台环境 / 产出物形式 | **E 交付形态** |
| 集成商等级 / 用户角色分级 / 权限分级 / 访问控制 | **F 用户/集成商分级** |
| API 数量上限 / 接口预算 / 复杂度约束 | **G API 预算** |
| 产品原则 / 设计准则 / 取舍优先级 / 破坏性变更审批 | **H 产品原则取舍** |

做法：① 没有本地档案就先建（脚本或复制模板），已有就**增量更新、不覆盖用户已填**；② 抽取后**向用户复述结构请其确认**（"我从你资料里抽出这套能力层级/域/实现层/命名约定，对吗？"）；③ 资料里 skill 用不到的敏感内容（未公开路线图、内部成本、个人隐私等）**不纳入**；④ 全程**脱敏红线**：真实内部接口清单/团队名/预算只落这份本地文件、`.gitignore`、绝不提交、绝不进公开产出。

## 模板（复制到 `product-profile.local.md` 后按你产品填）

> 下面全是**产品无关的占位**，每项给多领域示例帮你对照。`<…>` 处填你产品的真实定义。用不到的行删掉，缺的层级/域/实现层自行增补。**示例只是示例**——别照抄机器人那套，填你产品自己的。

### A. 能力层级（对外抽象分几层、各层是什么）
> 用你产品自己的分层名，`L1/L2/L3` 只是通用记号；层数不必是 3，能力型产品常见 2~4 层。

```
L1 = <最底层/最原子的对外能力>
        机器人: 原子能力 / 底层运动与感知接口
        SaaS:   底层 API / 单一资源的 CRUD
        芯片SDK: 寄存器/HAL 级原语
        云平台: 单一云服务的基础 API
L2 = <组合/应用级能力>
        机器人: 组合能力 / 应用级开发包
        SaaS:   业务能力组件 / 工作流编排
        芯片SDK: 驱动/中间件 SDK
        云平台: 托管服务 / 高阶封装
L3 = <完整方案/场景交付>
        机器人: 完整方案 / 场景交付
        SaaS:   开箱即用的行业解决方案
        芯片SDK: 参考设计 / 一体化方案
        云平台: 全托管解决方案 / 蓝图
（层数与命名按你产品改；也可只保留 2 层或扩到 4 层）
```

### B. 能力域（横向切成哪几个域）
> 域是产品能力的横向切分（"这个需求属于哪一块"）。数量按你产品定，不必是 5。

```
域1 = <…>   域2 = <…>   域3 = <…>   域4 = <…>   域5 = <…>
  机器人: 运动 / 感知 / 交互 / 数据 / 标定
  SaaS:   账号与权限 / 计费 / 数据 / 集成 / 通知
  芯片SDK: 外设驱动 / 电源与时钟 / 通信 / 安全 / 调试
  云平台: 计算 / 存储 / 网络 / 可观测 / 身份
  音视频SDK: 采集 / 编解码 / 传输 / 渲染 / 混流
```

### C. 实现层 × 团队（对内由哪些层承接、各层归谁）
> **这是最需要你按产品自定义的一项。** 与能力层级正交——能力层级管"对外抽象多高"，实现层管"对内动哪些技术栈、归几个团队"。**别硬套机器人的固件/算法/硬件**，填你产品自己的实现层。

```
实现层 → 负责团队/角色 owner → 跨层签字人
  <你产品的运行时实现层，逐层填>
    机器人示例: 二开 SDK 层→<SDK组>；底层软件(固件·中间件)→<固件组>；算法服务→<算法组>；硬件(机械·电子)→<硬件组>
    SaaS 示例:  前端→<前端组>；后端/服务→<后端组>；数据/数仓→<数据组>；基础设施/平台→<SRE/平台组>
    芯片SDK示例: 应用 SDK 层→<SDK组>；驱动/HAL→<驱动组>；固件/RTOS→<固件组>；硅片/IP→<硬件/IP组>
    云平台示例: 控制面 API→<API组>；数据面/运行时→<运行时组>；控制台/前端→<前端组>；底层基础设施→<基础设施组>
  非运行时层（生态/集成/文档/打包类特性会落这里，别硬塞给核心研发层）：
    文档/开发者门户层     → <如 DevRel/文档/站点前端> → <签字人>
    构建/分发基建层       → <如 构建/DevOps/发布工程>   → <签字人>
  跨团队升级路径 = <如 层 owner → 部门负责人 → 架构评审会 → CTO>
```

### D. 接口与命名约定（对外契约怎么写）
```
命名风格 = <如 接口/参数全小写下划线 snake_case、词意完整、禁用缩写(mic→microphone)>
          （或你产品的规范：REST 资源用复数名词 / gRPC 用 verb_noun / C API 用 前缀_模块_动作 …）
语言/绑定约定 = <如 C++ core + pybind11 / TypeScript SDK / REST+OpenAPI / gRPC+protobuf / C header>
错误与返回约定 = <如 失败返回 bool 不抛异常 / HTTP 状态码 + 错误体 / errno 风格 / Result 类型>
物理接口约定[仅硬件类产品] = <机械: 法兰/孔位/公差/承重/力矩单位；电气: 连接器/pinout/供电/协议>
```

### E. 交付形态（产出成什么、跑在哪）
```
形态 = <如 仿真/真机/两者 · SaaS 多租户/私有化 · SDK 包/源码 · 芯片评估板/量产 · 云 SaaS/on-prem>
支持平台/环境 = <如 x86+arm × Ubuntu18.04~24.04 / 浏览器矩阵 / MCU 型号 / 云 region>
产出输出目录 = <默认 feature-specs/；产出写到 <此目录>/<特性名>/<特性名>.md>
```

### F. 用户 / 集成商分级（对谁开放、按级门控）
```
分级 = <如 Associate / Professional / Expert · 免费/Pro/Enterprise · 只读/开发者/管理员 · 已购机客户/公开>
分级差异 = <各级能用哪些能力、对应封装/文档复杂度、危险/高权限动作对哪级开放>
```

### G. API 预算（接口规模纪律）
```
API 预算 = <如 300 个方法上限 / REST 端点数上限 / 每模块接口数软上限 / 无硬上限但需评审>
占用判定 = <新增接口如何计入预算、超阈值触发什么评审>
（预算是"控制对外接口面别失控"的纪律；无硬性上限的产品可写"无硬上限，新增需接口评审"）
```

### H. 产品原则取舍（产出前自检 gate 的尺子）
```
采用内置默认原则：是 / 否（默认=是，10 条全文见 references/product-principles.md；均领域中立）
团队自定义/补充原则：<可留空；填了则与内置默认合并，冲突以团队为准>
冲突优先级：<两条硬约束不可覆盖：安全默认、向后兼容均为红线，二者相撞必须升级人工拍板；
             其余原则相对优先序在此填，留空则相撞时抛回用户拍板，不自动排序>
破坏性变更签字人：<判定为 break 时需谁签字，如 平台架构负责人 / API 委员会>
```

## skill 如何使用本档案（落到产出的哪里）

细化需求、走到产出时，若检测到本地 `product-profile.local.md`，按它做归类、分解与产出（**复用现有节，不新增结构**）：

- **归类（第 3 步）**：用 **A 能力层级 + B 能力域**把需求钉到坐标系（落在哪层、属哪域、是否跨域）；无档案则声明"以下层级/域为**推断**"并给依据、打 `[待确认]`。若特性是纯交付/分发/安装/文档形态（非运行时能力），层级按 skill 规则标 `N/A`。
- **跨层分解与路由（第 4 步）**：用 **C 实现层 × 团队**逐层判定"要不要动这一层、归哪个 owner、谁先谁后"；owner/签字人按 C 落到真实团队，无档案则打 `[待确认·需指派]`，**绝不臆造团队名**（错派实现层是最危险的失败）。
- **接口契约草案**：套 **D 命名与语言/绑定约定**写接口签名、参数、错误返回；硬件类特性套 D 的物理接口约定。
- **影响评估 · 兼容 · 发布策略**：按 **G API 预算**判新增接口占用与是否触阈值评审；破坏性变更需 **H 的破坏性变更签字人**；发布策略的灰度/开关按 **F 分级**（对哪级集成商/角色开放）。
- **能力定位 / 范围边界**：交付形态、支持平台、目标用户级引用 **E 交付形态 + F 分级**。
- **产出前自检 gate**：按 **H 产品原则**逐条给三态结论（✓/⚠/—）；无档案则用 references/product-principles.md 内置默认 10 条，并注明"本次采用内置默认产品原则"。**gate 任何情况下都要跑，绝不因档案空着就略过。**

## 与配置区 A~E 的关系

SKILL.md《配置区》里的 A~E 组（A 能力坐标系 / B 接口约定 / C 集成商与形态 / D 产品原则 / E 实现层 × 部门）就是**产品档案的内联最小形态**——把它们从"写在 SKILL.md 里的占位符"抽出、一般化、可由脚本生成、可从产品资料自动映射，就是本档案。二者是**同一套坐标系的两种载体**：

- **配置区 A~E（内联）**：随手填在 skill 里，适合单一产品、少量字段、快速上手；本档案的 A(层级)+B(域) ↔ 配置区 A，D(命名) ↔ 配置区 B，E(形态)+F(分级)+G(预算) ↔ 配置区 B/C，H(原则) ↔ 配置区 D，C(实现层×团队) ↔ 配置区 E。
- **本档案 `product-profile.local.md`（外置）**：可生成、可脱敏外置、可从资料自动填、可 gitignore，适合内部信息多、要与开源 skill 分离、要团队共享的场景。

skill 读取优先级：**有本地 `product-profile.local.md` 时以它为准**（更完整、可含私有信息）；没有时回退到 SKILL.md 配置区 A~E 的内联值；两者都空时用内置通用坐标系占位并把推断打 `[待确认]`。填了本档案就**不必再维护 SKILL.md 里的配置区占位**——避免两处不一致。
