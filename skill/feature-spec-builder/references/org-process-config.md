# 组织与流程 · 本地配置（模板 + 机制）

## 这是什么、为什么放本地

feature-spec-builder 是**开源**的，但"**谁负责什么、要过哪些流程 gate、版本怎么排**"是**各公司私有**的信息，而且每家都不一样。所以这些**不写进 skill**（开源、脱敏、无人名），而是由你**在本地生成一份配置文件**填自己公司的情况，与 skill **搭配使用**。

- 公开的 skill 里：只有下面这套**通用占位模板**（角色名、流程阶段、gate 类型），**没有任何真实人名 / 内部链接**。
- 你本地填好的配置：含真实责任人、真实流程节奏——**本地私有、加入 `.gitignore`、绝不提交**。

## 怎么用（三步）

1. **生成本地文件**：跑 `scripts/init-org-config.sh`，或手动把下面《模板》整段复制成你工作目录下的 **`org-process.local.md`**。
2. **按你公司情况填**：角色地图填真实责任人、流程 gate 填你们实际的评审节点、版本节奏填你们的周期与容量。
3. **搭配 skill**：细化需求时，skill 会读这份本地配置——按角色地图**路由 owner**、列出该特性**必过的流程 gate 与交付件**、把特性**挂到目标版本与管道容量**、判断变更**是否要走 CCB**。**没有这份配置时**，skill 用通用角色占位（如 `[产品经理·待指派]`）并提示你生成一份。

> **隐私红线**：真实人名 / 内部 URL 只进 `org-process.local.md`（本地、gitignore）。skill 产出的特性需求文档若要公开或跨公司传阅，owner 用角色名或占位，不带真实人名（除非该文档本身是内部件）。

### B. 交付流程与关键 gate（5 阶段 —— 填你公司实际的 gate 与交付件）
1. **需求规划与澄清**：需求收集 → 澄清（交付件：**PRD**）→ 评估（**工作量估分·人天**）→ **需求锁定**（gate：对齐版本需求范围）→ 需求创建。
2. **需求设计与开发**：方案设计评审（交付件：**设计文档 + 测试用例**，测试参与评审）→ 日常构建质量 → **需求变更走 CCB** → 合入规范（保护分支 / Maintainer 审 MR）→ 开发 + 自测（gate：**自测报告 = 提测入口条件**）。
3. **测试验证**：分支拉取 + 自验证编译 → RC 构建 → 冒烟 / 集成 / 场景 / 压测（交付件：**测试报告**；质量指标按业务线定）。
4. **版本验收与发布**：需求验收（产品/提出人）→ **发版评审**（gate：综合质量结论给发版结论）→ 版本说明会 → 版本总结。
5. **版本生命周期**：导产验证、运维例会、**Hotfix**（严重且影响场景使用的问题，评审后增发）、**EOS**（维护窗口外不再响应）。

## skill 如何使用本配置（落到产出的哪里）

细化需求、走到产出时，若检测到本地 `org-process.local.md`，按它填《输出模板》的以下位置（**复用现有节，不新增结构**）：

- **owner 路由**：《跨层实现与跨部门协同》《遗留待确认》里的 owner，按 **A 角色地图**落到真实责任人；无配置则留 `[角色·待指派]`。
- **必过 gate 与交付件**：《影响评估 · 兼容 · 发布策略》里列出该特性要过的 gate（需求锁定 / 设计评审 / 自测报告 / 需求验收 / 发版评审）与对应交付件（PRD / 设计文档 / 测试报告 …）。
- **版本与容量**：《影响评估》里标注**目标版本** + **工作量估分（人天）vs 管道容量**（C）；大特性提示按大/小版本定位。
- **变更控制**：若需求已过"需求锁定"再变更，提示**走 CCB**（D）。
- **可选新增读者**：《阅读指引》可加"项目 / 质量 / 运维"等角色行（按 A）。

> 与"实现层 × 部门（配置区 E 组）"的关系：E 组是**技术实现层**的 owner（二开/固件/算法/硬件/文档门户/构建分发）；本配置是**产品与交付流程**的角色地图与 gate（产品/项目/质量/测试/运维/CCB/版本）。两者互补：E 组回答"哪层动、哪个技术团队做"，本配置回答"走哪套流程、过哪些评审、挂哪个版本、谁验收"。
