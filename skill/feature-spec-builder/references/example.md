## 一句话定义
提供一个面向 Professional 级集成商的导览能力包：机器人按预设/动态路径带领访客行进、全程避障，并在指定途经点触发由集成商提供的讲解内容。

## 来源与背景（Why）
- 原始诉求：客户希望有"迎宾导览功能"。
- 业务场景：展厅讲解导览为主，但能力包需通用，可复用于大堂接待、政务导引等。
- 为什么现在做：接待类应用里"接待+带路+到点讲解"是高频重复动作，今天每个集成商都要自己从底层导航/避障/定位拼一遍，重复造轮子且各家质量参差。
- 价值：把这套高频动作沉淀为可复用组件，降低集成商搭建接待类应用的成本、缩短交付周期，并把避障/安全等风险点收敛到一处统一保证。

## 名词解释（术语）
| 术语 | 说明 |
| --- | --- |
| 途经点（waypoint） | 导览路线上预先标好的一个个停靠点位（坐标，展厅场景还带朝向）；机器人按途经点序列依次走 |
| 到点回调（on-arrive callback） | 机器人到达某个途经点时，能力包主动回调集成商传入的一段函数，把"控制权"交给集成商去播讲解——讲什么由集成商决定，不在本包内 |
| 钳制（clamp） | 把集成商传入的参数值强制限制在安全范围内：超出上下限就取到边界值而非报错放行，同时告警。用于速度/避障距离这类越界会有物理风险的参数 |
| Professional 集成商 | 集成商分级中的一档：给"高层入口 + 关键参数可覆盖"，比 Associate（高度封装、几乎不可调）灵活，比 Expert（细粒度控制）安全边界更强；本包面向这一档 |
| L2 / L3 | 能力分层：L2 是给集成商二次开发用的"组合能力/开发包"，L3 是开箱即用、可直接部署的成品方案；本包是 L2 |
| 重定位 | 机器人在定位丢失后重新确定自己在地图中位置的过程 |
| DX（开发者体验） | 集成商开发者用这个包时的顺手程度：接口是否好懂、命名是否一致、报错是否能定位根因 |

## 能力定位
- 层级：L2（组合能力 / 应用级开发包）
- 能力域：跨域 —— 运动/导航域 + 感知域（途经点识别与避障）
- 跨域影响：依赖导航与感知两域的底层能力；讲解内容播放接口与音频/交互域对接但不属本特性范围。
- 跨层结论（第 4 步）：**单层特性（二开 SDK）** —— 在 SDK 层组合已有的导航/避障/定位能力，无固件/算法/硬件新建改动。

## 范围边界
- 包含（In Scope）：路径规划（基于预设途经点序列）、避障行走、到达途经点触发"讲解回调"、仿真与真机两套运行。
- 明确不做（Out of Scope）：
  - 讲解内容本身（讲什么）—— 由集成商通过回调传入
  - 多人接待 / 排队调度
  - 人脸识别身份核验
- 关联特性占位（来自需求分流、本轮不做）：无。

## 跨层实现与跨部门协同
单层特性（二开 SDK），无跨层协同。

## 开发者/集成商故事（按 P1/P2/P3 优先级）
- **[P1]** 作为集成商开发者，我希望用一个高层入口传入途经点序列即可启动导览，以便几行代码完成接待应用主流程。
  - 验收标准（含 DX 验收）：给定 N 个途经点，机器人按序到达全部点位、无碰撞、到点触发回调；接口命名/参数风格与同域既有导航接口一致，失败返回可定位根因的错误码。
  - 优先级依据：客户核心诉求（导览主流程），硬需求。
  - 仿真判据：样本 N=5 途经点 ｜ 前置：标注好途经点的仿真场景地图 ｜ 判定：到达率与碰撞由仿真器轨迹日志判 ｜ 阈值：到达率 100%、无穿模碰撞。
  - 真机判据：样本 N=5 ｜ 前置：对应实景地图 ｜ 判定：定位系统读数 + 人工观察 ｜ 阈值：到达定位误差 ≤ [待确认] cm、全程无接触式碰撞。
- **[P1]** 作为集成商开发者，我希望在到达每个途经点时收到回调，以便插入自己的讲解逻辑。
  - 验收标准：每个途经点到达后回调被调用且携带点位标识。
  - 优先级依据：与导览主流程同等核心，硬需求。
- **[P2]** 作为 Professional 集成商，我希望能覆盖关键参数（行进速度、避障距离阈值），以便适配不同场地。
  - 验收标准：传入自定义参数后行为按预期变化；超出安全范围的值被钳制并告警。
  - 优先级依据：适配性增强项。

### MVP 最小切分
- 本轮最小可交付集(MVP) = 两条 P1（按序到点导览 + 到点回调）。
- 砍到 MVP 后是否仍对客户成立：是 —— 展厅讲解主流程靠这两条即可闭环。
- 明确推迟到下一迭代：P2 参数覆盖（默认参数先够用）。

## 行为规格（Behavioral Spec）
- 正常流程：接收途经点序列 → 规划路径 → 依次行进（全程避障）→ 到达每点触发讲解回调 → 全部完成后结束会话。
- 状态与生命周期：`空闲 → 行进中 →（暂停）→ 行进中 → 结束/中止`。支持 pause/resume；暂停态拒绝新的 start 请求并返回明确错误码；stop 立即结束并复位到空闲。
- 异常与失败处理：
  - 路径被堵：原地等待 10s（默认，可调）→ 仍堵则尝试绕行重规划 → 绕不过则上报并中止。
  - 定位丢失：暂停行进并上报，触发重定位流程（[待确认] 是否本包负责）。
- 边界与边缘场景：途经点序列为空 → 拒绝并报错；单点 → 走到即结束；中途 stop → 安全停车并复位。
- 参数细则：行进速度（默认/范围 [待确认]，覆盖值钳制在安全范围）、避障安全距离（默认/范围 [待确认]）、堵塞等待超时（默认 10s，可调）。
- 时序与性能：到点回调需在到达后 ≤ [待确认] ms 内触发。
- 并发与可重入：同一时刻仅允许一个导览会话；行进态/暂停态收到新 start 一律拒绝。
- 可观测性：面向调用方[当前状态查询、当前途经点序号、到点/异常事件回调] ｜ 面向平台运营[导览成功率、中止率、平均完成时长埋点]。

## 非功能性需求(NFR) 与上线就绪
- 安全与权限：对 Professional 级开放；自主移动属有风险动作，速度/避障距离的覆盖值必须钳制在安全范围、越界告警；保留急停优先级最高。
- 数据与隐私/合规：本特性不采集个人/敏感数据（途经点为预设坐标，讲解内容由集成商提供）。
- 运营可观测：上线后看导览成功率/中止原因分布/采纳集成商数；中止率突增告警。
- 性能与容量[按需]：避障实时性与到点回调时延 [待确认]，由算法同学给基准。

## 接口契约草案（签名级，不写实现）
- 新增接口（示意）：
  - `start_tour(waypoints, on_arrive_callback, options?)` ｜ 输入：途经点序列、到点回调、可选参数 ｜ 输出：导览会话句柄 ｜ 错误码：路径不可达 / 途经点非法 / 会话已存在 ｜ 异步
  - `pause_tour(handle)` / `resume_tour(handle)` / `stop_tour(handle)`
  - `get_tour_status(handle)` ｜ 输出：状态 + 当前途经点序号
  - `options` 可覆盖：行进速度、避障距离阈值、堵塞等待超时
- 命名与绑定约定：全小写下划线、词意完整、禁用缩写；与同域既有导航接口风格一致；C++ core + pybind11，回调遵循现有事件回调风格。

## 测试 / QA
> 测试对象是**运行时能力**：机器人是否按序到点、全程避障、到点如实回调，以及对外接口/错误码/参数钳制是否可测、可自动化。**仿真先行**——验收先在仿真里跑通再上真机。判据的样本量与阈值见各故事的"仿真判据/真机判据"。
- **要覆盖的场景/矩阵**：
  - 正常流程：N 个途经点按序全部到达、全程无碰撞、每点触发一次回调（仿真 N=5 起，见 P1 判据）。
  - 状态机：`空闲→行进中→暂停→行进中→结束/中止` 每条转移；暂停态发新 start 必须被拒并返回明确错误码；stop 后复位到空闲。
  - 异常：路径被堵（等待 10s→绕行→绕不过中止 三段各验一次）；定位丢失（暂停+上报）。
  - 边界：途经点序列为空（拒绝报错）、单点（走到即结束）、中途 stop（安全停车+复位）。
  - 参数钳制：越界的行进速度/避障距离被钳制到安全边界并告警，而非放行或直接报错。
  - 运行形态矩阵：仿真 + 真机两套都要过（同一批场景各跑一遍）。
- **对外契约是否可测**：
  - 接口层：`start_tour / pause_tour / resume_tour / stop_tour / get_tour_status` 的返回值与错误码（路径不可达 / 途经点非法 / 会话已存在）逐个可断言。
  - 回调层：到点回调被调用的次数、时机、携带的点位标识可被测试桩捕获校验；到点回调时延 ≤ [待确认] ms 可测。
  - 可观测层：状态查询、当前途经点序号、成功率/中止率埋点可读，供测试判定与运营核对。
- **判据归属**：到达定位误差/朝向阈值、避障安全距离、回调时延上限等阈值现为 [待确认]，由算法同学给基准后，QA 据其定量验收（见《遗留待确认》）。
