# 跨层路由与跨部门协同（第 4 步用）

很多客户需求看似「二开加个接口」，实则要动二开层以下的层——固件、算法、甚至硬件，而这些层往往归不同部门。本文件是 SKILL.md 第 4 步「跨层分解与路由」的操作手册：怎么判一个需求落在哪几层、怎么排依赖顺序、怎么把层间交接和跨部门协同写进文档、怎么在「下层做不出来」时收口、跨层时怎么在**一份 md 内分阅读对象**组织（一个特性只出一份文档）。

全程严守《交互式澄清规则》：一次一问、带候选、不替用户拍板、推断项打 [待确认]。

> 贯穿例子：客户要 `set_mic_beam / get_mic_beam`（设置/读取麦克风波束方向）。**按命名规范（全小写下划线、词意完整、禁用缩写）标准化为 `set_microphone_beam_angle / get_microphone_beam_angle`**（`mic`→`microphone`，补 `angle` 自解释；重命名记入《关键决策记录》，来源=产品原则第 5 条）。表面是二开层加一对 set/get，实则强依赖固件是否已支持波束成形；若固件没有，级联到算法（波束成形算法）甚至硬件（麦克风阵列阵型）。

## 一、实现层模型（4 层）

实现层与 L-Tier **正交**：L-Tier 回答「对外抽象多高」，实现层回答「对内要改哪些技术栈、动几个部门」。同一个 L1 接口，可能只动二开层，也可能一路动到硬件。

| 层 | 职责边界 | 典型 owner 部门 | 改动周期 |
| --- | --- | --- | --- |
| 二开 SDK 层 | 对集成商暴露的二次开发接口/能力包；透传或适配下层能力 | SDK/平台组 | 短 |
| 底层软件(固件·中间件) | 固件、驱动、中间件、实时控制；直接驱动硬件、暴露底层能力给上层 | 嵌入式/固件组 | 中 |
| 算法服务层 | 感知/规划/标定/模型推理等算法能力 | 算法组 | 中~长（含数据/训练/标定） |
| 硬件层 | 机械结构、电子、传感器、线缆布局 | 硬件/电子组 | 长（涉及打样/选型/改板） |

> 各层真实部门名以配置区 E 组「实现层 × 部门映射」为准；未填时按上表推断，但推断的 owner 一律打 `[待确认·需指派]`，绝不臆造具体部门名。若团队实际技术栈不止这 4 层（如有云端服务层/仿真层），在配置区 E 组补行并相应扩展分解表。

> **除运行时 4 层外，还有两类非运行时层**（生态/集成/文档类特性——detail-probes 原型 H——重心常在此，见配置区 E 组）：**文档门户层**（= 文档/开发者门户；官网板块、示例仓、文档 IA；owner=DevRel/文档/站点）与**构建/分发基建层**（打包、多环境 CI、ABI/预编译；owner=SDK 构建/DevOps）。交付物是示例/打包/文档时，即使技术触底=纯二开，也路由到这两层的 owner，**别默认派给二开 SDK 组**。

### 外部依赖轴（与实现层正交，四层模型接不住的第五类）
有的需求依赖**外部框架/生态/OS/语言运行时**（如 ROS distro、DDS、Ubuntu、CPython、第三方云服务）。它们的排期与生命周期**不归任何内部部门**，"找部门确认"收不了口——由外部社区决定。判定与处置：
- **命中判定**：需求是否依赖外部框架/生态/OS/运行时？（集成/适配/绑定类几乎必命中。）
- **强制问三条**：① 外部依赖的**版本支持窗口 / EOL 时钟**归谁跟随（如 ROS Noetic 已 2025‑05 EOL）？② **兼容矩阵回归成本与 CI 归属**？③ 外部 break/EOL 时的**降级策略**？
- **"外部依赖待确认"（区别于跨部门待确认三元组）**：外部社区不会给你私有排期答复，故默认按**保守假设**走——"外部随时可能 EOL/break"，并定一条**长期维护承诺分级 + EOL 跟随策略**（跟随最新 LTS？支持 N 个版本窗口？谁在升级时更新并跑 CI）。
- 与 detail-probes 原型 H 探针②（版本矩阵+EOL）、④（示例防腐）交叉引用。

### 当「构建/分发基建层」是最底改动层（打包/分发/安装类主特性，原型 I）
离线安装包 / 安装器 / 部署包这类特性，最底改动层就是**构建/分发基建层**本身（不是被顺带点名的附带层）。此时：
- **触底结论 = A/B**，**不下探固件→算法→硬件**（能力已存在，是把既有产物打包分发）；用"跨职能 owner（构建/DevOps + 文档门户）+ 外部依赖轴"视角，别套原型 G 的"底层支不支持"。
- 探针清单见 detail-probes **原型 I**；其产物落点复用现有输出模板节：平台矩阵+依赖捆绑→《依赖与前置》《影响评估》；完整性签名+权限落点→《NFR·安全与权限》；幂等/升级/卸载/回滚→《影响评估·兼容·发布策略》；失败可诊断+README→《行为规格·可观测性》+ P1 的 DX 验收。
- **⚠ 高危错派红旗（与"示例/文档默认派二开组"并列）**：**打包/分发/安装类默认被派给二开 SDK 组**。这类 owner=构建/DevOps（+文档门户），二开组仅出"装后 import+冒烟"验证点，绝不承接整条需求。

## 二、逐层判定问题（判断需求落在哪几层）

对每一层各问一遍「要不要动、动什么」，自顶向下问、自底向上验：

**二开 SDK 层**
- 对外要新增/修改什么接口或能力包？
- 二开层是**原样透传**下层能力，还是要做单位/坐标系/枚举的**适配转换**？（纯透传≈下层必须已具备该能力）

**底层软件(固件·中间件)** ← 跨层需求最常见的引爆点
- 二开要暴露的能力，固件**已经有、只是没开接口**，还是**根本没实现**？（前者「开个口」，后者「从零做」，量级差一个数量级——必须固件 owner 确认，不能替答）
- 需要固件向上暴露什么新接口/数据/事件？时序、实时性要求？

**算法服务层**
- 这个能力本质是不是算法问题（波束成形/降噪/检测/SLAM/标定）？算法**是否已就绪**？
- 需不需要新的训练数据、标定流程、模型？（会显著拉长周期）

**硬件层** ← 最该早识别，周期最长
- 该能力是否受硬件形态物理限制？（如波束成形依赖麦克风阵列的元数与阵型；某些角度硬件做不出）
- 是否需要改板/选型/重新布局？现有硬件能不能支撑标称指标？

**判定产物**：一张跨层分解表（层｜要改什么｜是否真要动｜owner 部门｜阻塞关系）+ 一个「最底改动层」。

> set_microphone_beam_angle 逐层走一遍：
> - 二开层：加 `set_microphone_beam_angle/get_microphone_beam_angle`，**透传**固件波束控制。→ 要动，但只是薄封装。
> - 固件层：固件**是否已支持波束成形并可调方向**？→ 若否，这才是真正的工作主体。**必须找固件组确认**。
> - 算法层：波束成形算法是否就绪、是否需要按阵型重新设计？→ 视固件答案而定。
> - 硬件层：当前麦克风**是不是可成形的阵列（几元/线阵/环阵）**？→ 若是单麦，波束成形物理上无从谈起，需硬件改型。

## 三、「非二开能独立交付」的硬 gate + 可行性证伪 + 结论 B/C 回写

找出需要改动的**最底层**。一旦某层答「不支持/不确定」，沿 二开→固件→算法→硬件 逐层下探，直到**触底**——找到能支持的层，或判定**物理不可行**。

**层级触底结论（写进《能力定位》节，三选一）：**
- **A. 纯二开层可交付**（下层已具备所需能力）→ 普通二开特性，正常走后续步骤。
- **B. 跨层可交付**（需 X 层先做 Y，见依赖顺序与协同视图）。
- **C. 当前层级不可交付**（受下层未排期/物理限制阻塞，见跨层阻塞与降级）。

最底改动层 ≠ 二开 SDK 层 → **本特性不是二开层能独立交付的**，在文档显著位置打标：
> `⚠ 非二开层可独立交付：本特性最底改动落在 <层>，需 <owner 部门> 先交付 <能力>，二开层才能封装。`
并写进《能力定位》节与 DoR。

- **触到硬件/物理限制时，立即停止在二开层细化接口签名与参数。** 继续把它当二开需求往下写接口契约草案，产出的是一份永远做不出来的壳，必被评审拒稿。转去填《跨层阻塞与降级》。

### 结论 B/C 必须回写主叙述（堵死「假就绪」）
一旦结论为 B 或 C，**必须回改输出模板《一句话定义》与 P1 故事**，使其只描述本轮**真实可交付物**，而不是客户原始诉求的完整能力：
- 反例（假就绪）：一句话定义写「设置麦克风波束方向」，P1 写「调 set_microphone_beam_angle 即可指向任意角度」——但固件根本没有波束成形。
- 正例（收口）：一句话定义写「在 SDK 层提供 set/get_microphone_beam_angle **接口桩与签名**，波束实际能力依赖固件波束成形（未排期，最早 Q4）」，P1 只承诺「桩可联调、签名稳定」。
- **DoR 红线：结论 B/C 而 P1 仍承诺下层未就绪能力 = 不过 DoR。**

### 结论 C 且二开层本轮无任何可交付物（含桩）→ 早停
若证伪确认本特性当前实为下层/硬件改动、二开侧连桩都没意义（如物理不可行、或下层接口形态完全未定导致桩都无从写起），**不要继续走第 5、6 步逐维澄清**。直接产出一份**精简「跨层路由结论」**：
```
# 跨层路由结论：[特性名]（非二开层任务）
- 层级触底结论：C —— 当前不可二开交付
- 跨层分解表：[见下]
- 关键阻塞与升级路径：[谁阻塞、找谁、何时复评]
- 给客户的口径：[当前能/不能交付什么，预计何时具备]
- 结论：本需求当前不是二开层任务，建议转 <部门> 立项；二开层在 <下层能力> 就绪后再启动封装。
```
不再套完整输出模板，避免产出满是 [待确认] 的空壳、浪费交互预算。

> set_microphone_beam_angle：若固件确认「尚不支持波束成形」，最底改动层至少到固件层（甚至硬件层），结论是 B 或 C，绝不是「二开加两个方法」。若连对外是角度还是档位都取决于尚未启动的硬件选型，则连桩签名都未定，走早停精简结论。

## 四、跨层依赖排序指南

跨层依赖几乎总是**自底向上**：下层就绪是上层启动的前置。
- **画依赖链**：[硬件/固件] → [算法] → [二开]，箭头表示「先就绪才能启动」。
- **标关键路径**：最可能卡住整体进度的那一环（通常是周期最长或最不确定的下层）。
- **区分可并行 vs 必须串行**：文档/示例/对外接口设计/桩可与下层开发并行；但联调与对外验收必须等下层就绪。
- **每条依赖标确认状态**：「已确认支持 / 待 X 部门确认 / 确认不支持需新建」。

> set_microphone_beam_angle 依赖链：麦克风阵列（硬件，确认阵型）→ 固件波束成形可调（固件）→ 波束控制接口暴露（固件/算法）→ `set/get_microphone_beam_angle` 封装（二开）。关键路径阻塞点：「固件是否支持波束成形」——未确认前，二开层只能做接口设计与桩，不能联调验收。

## 五、层间交接契约（Handoff）模板

每条跨层依赖配一份 handoff，把「下层向上层暴露什么」讲死，并钉死**定义权归属**（避免二开按 `set_microphone_beam_angle(angle_degrees)`、固件按 `set_beam(index)` 各写一版，集成时对不上、评审互相甩锅）：

```
[下层 → 上层] <能力名> handoff
- 定义方（拥有定义权的层/部门）：未达成一致则标 [待确认·需 X 与 Y 对齐]
- 下层暴露：接口/数据/能力（签名级）
- 输入：类型 / 单位 / 范围 / 坐标系 / 默认值
- 输出：类型 / 单位 / 含义
- 非法输入语义：拒绝返回错误 / 钳制 / 就近吸附；失败时原值是否保持
- 标称范围 vs 实际档位：对外声明范围 与 底层真实可达分辨率/档位 是否一致；不一致如何映射
- 单位/坐标系换算责任：上层透传 / 上层转换 / 下层适配
- 同步/异步、生效时机、掉电是否保留
- 可观测性：配套的 get/查询/事件，让上层确认下层当前状态
```

> set_microphone_beam_angle 的 [固件 → 二开] handoff：
> - 定义方：[待确认·需 SDK 与固件对齐：对外用角度还是档位]
> - 下层暴露：`firmware_set_beam_angle(angle_degrees)` / `firmware_get_beam_angle()`（示例签名，遵循无缩写命名）
> - 输入：角度，单位度，范围 [-90,90]，机体坐标系，默认 0（正前）
> - 非法输入语义：超范围**就近吸附到最近可达档位**，原值保持（需固件确认）
> - 标称范围 vs 实际档位：对外声明连续 [-90,90]，固件实际仅可分辨有限档（如 15° 一档）→ 决定「声明全范围+内部就近映射」还是「明示有限档」
> - 换算责任：二开层透传，单位与坐标系不转换（需与固件对齐确认）

## 六、跨部门协同视图（含跨部门待确认三元组）

直接填 SKILL.md 输出模板的《跨层实现与跨部门协同》节（跨层分解表 / 依赖链 / handoff / 协同治理轻表 / 跨层阻塞与降级 / 各层各自验收）。本文件第二~五节是素材来源：
- 跨层分解表 ← 第二节逐层判定
- 依赖链与关键路径 ← 第四节排序
- handoff ← 第五节模板
- 各层各自验收 ← 每层定义一条「可独立验证」的判据（固件层：示波/日志确认波束角可调；算法层：信噪比提升达标；硬件层：阵列拾音方向性测试达标；二开层：set 后 get 回读一致、对外接口契约符合）

**协同治理默认只填轻表**（各层 owner + 跨层签字人 + 升级路径）；RACI 整表仅在确需多部门并行执行、责任易扯皮的大型跨四层需求按需展开，否则不强制（多数格子会是 [待确认]，信息密度低）。

**跨部门待确认项必须带归属三元组（这是不让摩擦只被记录、而是被推进的关键）：** 使用者通常是产品/SDK 侧，会话里拿不到下层部门的真实答复，所以每个「某层是否支持/要不要动」的待确认项不能只丢清单，必须写：
- **验证 owner**：谁去对接该部门拿答案；
- **需在哪个评审节点前有答案**：deadline 锚到具体评审；
- **未及时答复的默认假设/降级**：在拿到答案前按什么保守假设走（如「假设固件不支持，先按 C 结论出桩+精简结论」）。

**跨层阻塞与降级** ← 下层「不支持且未排期」时，必须给 mock 桩 / 有限档位 / 仿真先行等降级或分期方案，并标 owner、预计解除、升级路径；不能只把硬阻塞塞进「遗留待确认」当无限期待办。

## 七、用法提示与红线

- 这一步**不替用户/部门拍板**：「要不要动固件」「硬件要不要改型」是涉及成本与排期的取舍，必须列成待确认项找对应部门确认，不能自己拍。
- 跨部门待确认项往往是整个需求的关键路径阻塞点，**优先把它们问清**——它们决定这份需求是「一周的薄封装」还是「跨季度的跨部门工程」。
- 推断的 owner 部门、未经对应部门确认的「是否支持」，一律打 `[待确认]`，写进《遗留待确认》节，并带三元组。
- **跨层即范围风险**：一个 PR 不该同时改四层，二开层不该越界替固件做本属固件的事。原则自检（第 7 步）的「单一职责/范围纪律」在跨层语境下就是「是否已按层拆 owner、是否存在某层越界承担他层职责」。

## 八、端到端示例：set_microphone_beam_angle（看似二开、实为跨 2~4 层、撞固件不排期）

### 输入
> 客户说：能不能给 SDK 加一对 `set_mic_beam` / `get_mic_beam`，让我设置麦克风波束方向。
> （按命名规范标准化为 `set_microphone_beam_angle` / `get_microphone_beam_angle`，重命名记《关键决策记录》，依据=产品原则第 5 条最小惊讶/DX。）

### 澄清片段（一次一个，证伪优先）
- 复述确认：你要的是在 SDK 层暴露一对接口，设置/读取麦克风阵列的波束指向，对吗？→ 对。
- 归类（第 3 步）：看似原型 G（接口暴露/薄封装）。**但进第 4 步可行性证伪前，不许当二开层定稿。**
- 证伪 Q1（向固件团队）：固件当前**支不支持波束成形**？→ 答：不支持，麦阵驱动只做了全向采集。
- 证伪 Q2：那是固件能加，还是要算法/硬件？→ 答：固件可加波束控制寄存器，但**波束成形算法**要感知算法组从头做；且现麦阵只有 2 阵元，**物理上只能做有限档位的粗波束**，要连续精确指向得换 4+ 阵元麦阵（硬件改动）。
- 证伪结论：跨 2~4 层（二开+固件+算法，精确版还要硬件），**非二开层可独立交付**。层级触底结论 = C。
- 阻塞 Q：固件波束控制本季度排期？→ 答：本季度不排期，最早 Q4。
- 早停判定：固件可加寄存器、对外档位形态可先定 → 二开层尚有「桩+签名」可交付，**不触发早停**，走完整模板但 P1 收口为桩。（若连对外形态都取决于未启动的硬件选型，则触发早停、只出精简「跨层路由结论」。）

### 产出（仅列与单层模板不同的关键节）

**能力定位**
- 层级：L1/L2 边界（底层原语+薄封装）
- ⚠ 层级触底结论：**C —— 当前不可纯二开交付。** 依赖固件波束控制（未排期）+ 算法波束成形（未启动）；连续精确指向受硬件阵元数物理限制。

## 九、一个特性一份 md：分阅读对象组织

一个特性**只产出一份特性需求文档**——哪怕跨层/跨部门，也不拆成多个文件。让不同团队都能在这一份 md 里追踪的办法是**分阅读对象**：文档顶部放一张《阅读指引》，《跨层实现与跨部门协同》节里各层内容按【阅读对象：X组】分节。这样一份需求全程在一处可追踪，各团队又能直接跳到自己那部分。

> （"多个特性各自独立成文"仍然成立——那是特性**之间**；本节说的是一个特性**内部**不拆文件。）

### 9.1 文档顶部《阅读指引》模板
（放在状态行之后、正文之前；纯单层小特性可省略）
```markdown
## 阅读指引（分阅读对象）
| 阅读对象 | 重点读哪些节 | 你负责什么 |
| --- | --- | --- |
| 产品 / 评审 | 定义、背景、定位、范围、故事+验收、影响评估、原则自检、决策、遗留 | 范围/优先级/取舍 |
| 二开 SDK 组 | 接口契约、行为规格、协同·二开节 | 对外封装与契约 |
| 固件组 | 协同·固件节、层间交接契约 | 确认/开放下层能力 |
| 算法组 | 协同·算法节 | 算法就绪/适配 |
| 硬件组 | 协同·硬件节 | 硬件形态/物理边界 |
| 测试 / QA | 各故事验收标准、对外契约（接口/物理契约）、平台/场景矩阵 | 照验收标准与契约设计测试、覆盖矩阵、验对外契约可测 |
```
（角色按特性类型选；**测试/QA 始终纳入**。不涉及的层写"本特性无需改动"或删去该行。）

### 9.2 《跨层实现与跨部门协同》里每层的分节模板
每个需改动的层，在协同节内放一个如下小节（贴上阅读对象标签，让该团队一眼找到自己那块）：
```markdown
### 【阅读对象：<X组>】<实现层>
- 本层职责：[这一层要交付什么，一句话]
- 上游依赖：[依赖哪层先交付什么；无则"本层是最底改动层"]
- 交接契约（handoff）：[引用本文件第五节；我从下层拿什么 / 我向上层暴露什么，签名级 + 定义权归属]
- 标称范围 vs 实际能力边界：[本层真实可达分辨率/档位/精度，如实]
- 本层行为规格：[与本层相关的正常流程/状态/异常/参数/时序/并发]
- 本层验收（可独立验证）：[不靠整体联调就能验的判据]
- 本层依赖与前置：[是否需新建能力/数据/标定/改板 + 量级；排期/阻塞]
- 本层待确认（带三元组）：[ ] 项 ｜ 验证 owner ｜ 评审节点前给答案 ｜ 未答复默认假设
```

> 与旧"主档案 + 子档（多文件）"的区别：**不再另起文件**。各层的 handoff、待确认、验收都收进同一份 md 的对应【阅读对象】节；《跨层分解表》的 owner 列指向本文档内锚点（而非外部文件）。要单独发给某部门时，把对应【阅读对象】节连同《阅读指引》复制出去即可，但**主追踪始终在这一份 md**。
